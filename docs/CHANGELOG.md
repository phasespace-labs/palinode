## Unreleased

## [0.9.3] — 2026-07-18

### Deprecated

- **`/save` and `/ps` slash commands — `/wrap` is the sole scaffolded lifecycle command (#631).** `palinode init` no longer writes `.claude/commands/{save,ps}.md` or their skill variants; `--slash` / `--skills` install `/wrap` only, and the scaffolded CLAUDE.md block now directs mid-session checkpoints to the `palinode_save` tool directly (`type="ProjectSnapshot"`) instead of a dedicated command. The lifecycle story simplifies to: deterministic SessionStart/SessionEnd hooks bracket the session, `/wrap` is the one human-typed command (session-end with push), and everything else is agent tool calls. **Nothing breaks for existing installs**: previously scaffolded `save.md`/`ps.md` files keep working and are never deleted or rewritten by a re-run, and the `palinode_save` MCP tool, `POST /save`, and the `palinode save --ps` CLI flag are all untouched — this deprecates the slash-command surface only. The `SAVE_COMMAND_BODY`/`PS_COMMAND_BODY` constants are removed; new regression guards assert init never scaffolds the deprecated commands and never clobbers legacy ones.

### Added

- **MCP registry publish is automated on tag push (#315).** A new public-repo workflow, `.github/workflows/publish-mcp-registry.yml`, fires on every `v*` tag: it waits for the tagged version to be live on PyPI (the registry entry references the PyPI package; `publish-pypi.yml` runs on the same tag), installs `mcp-publisher`, authenticates via GitHub OIDC (no stored secret), and publishes `server.json`. This closes the manual post-release registry step that let the registry drift to v0.8.6 metadata while PyPI was at v0.8.8. It runs as a separate workflow so a registry outage never blocks the PyPI release — a failed run is rerunnable via `workflow_dispatch`.
- **Session-start context for MCP-only clients — `/context/prime` goes live (#262, ADR-012 Layer 4).** Claude Desktop, Codex CLI, and Gemini CLI give Palinode only an MCP connection — no instruction file, no skills, no hooks — so memory was invisible there until the user typed a search by hand. Two levers close the gap, both spec-native. **(1) `instructions` on the MCP `initialize` response**: every client now receives a short, content-free memory contract (call `palinode_session_init` at conversation start, search before answering questions about prior state, save decisions, session-end at the end) — no memory content, so no scope risk, and it works on harnesses with no CWD. **(2) `palinode_session_init`** — the 28th MCP tool and the ADR's "first call you should make": returns a bounded digest of the resolved project scope — `core: true` memories, recent Decisions, open ActionItems — built from frontmatter reads and mtime sorts only (no embeds, no LLM), safe on the cold-start path. Scope discipline is strict: explicit `project` wins, else `cwd` resolves through the same ADR-008 `project_map`/`auto_detect` path the ambient search boost uses, and with neither the digest degrades to core-memories-only, clearly labelled — it **never guesses a project and never bleeds another project's context**. Harness policy: `auto_inject.harnesses_disabled` (default `["claude-code"]`) suppresses the digest for clients whose instruction-file/skill/hook layers already cover session start, detected via the initialize handshake's `clientInfo`; `auto_inject.enabled` and `auto_inject.instructions_enabled` are the kill switches. Per ADR-010 the capability lands on all surfaces: **`POST /context/prime`** — the endpoint the #261 SessionStart hook has been POSTing forward-compat since it shipped, now live with **no hook re-install needed** — and the new **`palinode prime`** CLI command (plugin-exempt). Registered in `palinode/core/parity.py`; MCP schema + API route-inventory goldens regenerated; tool tables updated to 28. New `tests/test_context_prime_262.py` (20 tests: resolution precedence, no-bleed project filtering, bounded output, closed-item exclusion, hook-shaped REST call, MCP schema/threading/suppression policy, initialize instructions, CLI rendering).
- **`palinode init` installs the palinode-session skill into every harness skill path (#264, ADR-012 Layer 2).** The canonical ambient-memory skill (`skill/palinode-session/SKILL.md` — recall at session start, save milestones/decisions with rationale, 30-minute progress notes, structured session-end capture) is now laid down by `init` instead of requiring a manual copy: `.claude/skills/palinode-session/SKILL.md` always, plus `.cursor/skills/` (Cursor) and `.agent/skills/` (Antigravity) when those harness footprints exist. `--user` installs to `~/.claude/skills/` (per-user, every project) instead of the project-scoped paths; `--skill-path <dir>` overrides the install root; `--no-skill` skips. The skill body is embedded in the CLI verbatim (it carries its own frontmatter) with a byte-for-byte drift-guard test pinning the embedded copy to the canonical repo file — same discipline as the session-start hook embed. Overwrite semantics are deliberately conservative: an existing SKILL.md is skipped unless `--force` (customizations survive re-runs), and a **symlinked** SKILL.md is never touched even with `--force` — a symlink means an external repo curates the real file, and breaking that ownership silently is worse than skipping. Nine new tests in `tests/test_cli_init.py` (drift guard, default install, harness detection, `--user`/`--skill-path`/`--no-skill`, customization preservation, symlink protection, dry-run listing).
- **`palinode init` scaffolds memory instructions for Cursor and AGENTS.md-aware harnesses (#263, ADR-012 Layer 1).** The memory-instruction lever now reaches beyond Claude Code: when the target project carries an `AGENTS.md` (or a `.agent/` directory — the Antigravity footprint), `init` appends a **harness-neutral memory block** to `AGENTS.md` (read by Codex, Antigravity, and other `AGENTS.md`-aware agents); when a `.cursor/` directory exists it writes `.cursor/rules/palinode.md` (a palinode-owned rules file) for Cursor. Detection-based defaults keep bare projects clean — nothing is written where no harness footprint exists — with explicit `--agents`/`--no-agents` and `--cursor`/`--no-cursor` overrides mirroring the `--claudemd` pair. The block is factored from a **single shared source** (`MEMORY_BLOCK_CORE`): the same recall / save-with-rationale / session-end contract CLAUDE.md carries, minus the Claude-Code-only machinery (`/clear` semantics, `/wrap`, SessionEnd-hook notes) that doesn't exist on those harnesses — one source inherited everywhere, no per-surface forks to drift. `CLAUDE.md` output is **byte-identical** to before the refactor, pinned by new golden fixtures (`tests/fixtures/claude_md_memory_block_{light,heavy}.txt`). All writers share the marker-idempotent create-or-append idiom — a user's existing `AGENTS.md` is never clobbered, and re-running `init` leaves exactly one `## Memory (Palinode)` section per file. New tests in `tests/test_cli_init.py` (byte-identity, core neutrality, detection defaults, explicit force/skip flags, append preservation, idempotency, dry-run listing); README "Daily use" section documents the new surfaces.
- **`unverified` — a 4th settable epistemic value: asserted but not checked (#589, ADR-018 amendment).** The epistemic vocabulary gains the state that previously had no honest home: a positive claim the writer is making while being explicit that it was never checked — neither verified into `fact` (which would overclaim), nor derived as an `inference` (which would mislabel it), nor an `open_question` (it isn't a question). Until now such claims fell through to `unmarked`, conflating "claim made but untested" with "no claim made at all" — a distinction that matters for an audit-grade store. Settable on all four surfaces per ADR-010 parity (MCP `palinode_save` enum, REST `SaveRequest.epistemic`, CLI `palinode save --epistemic unverified`, plugin via the shared contract; single source of truth stays `VALID_EPISTEMICS` in `palinode/core/parser.py`, so REST validation, metadata-dict tunneling rejection, and sticky carry-forward all inherit the new value with no additional wiring). Recall surfacing matches `inference`'s lower-trust treatment: MCP search results label hits `[unverified]` (with `fact` still unlabelled to avoid noise) and the `/ui` provenance **Claim type** row renders "unverified — asserted, not checked". An `unverified` claim is the prime candidate to acquire a `backed_by` evidence link and graduate toward `fact`. `unmarked` remains the non-settable absence default. ADR-018 §2/§4 updated (the alternatives note that originally deferred this value now records its addition); MCP schema golden regenerated; round-trip/provenance/search-label/parity tests added to `tests/test_epistemic_markers.py` and `tests/test_mcp_parity.py`.
- **Scoped memory selection — ADR-009 Layer 1 slice 3 (#107).** The listing path behind `GET /list` is now scope-aware: a new shared helper (`collect_memory_files`) accepts an optional `ScopeChain` and drops files whose **explicit** `scope:` frontmatter is not on the session's chain, and a pure predicate (`palinode.core.scope.chain_allows`) carries the visibility rule. Only explicit scope isolates — files without a `scope:` field behave exactly as before, in every mode (the ADR-009 §7 migration promise: "no scope = works as before"); the directory-inferred default stays reserved for the visibility/access layer (#108). `/list` rows gain a `scope` field (the explicit scope or `null`) so scoped setups can see why a file is or isn't selected; the `/list` HTTP contract is otherwise unchanged and never chain-filters — scoped selection is consumed by the upcoming `/context/prime` endpoint (slice 4). New `tests/test_scope_filtered_listing.py` (14 tests: predicate semantics incl. empty-chain and malformed-scope handling, helper filtering + `core_only` composition, `/list` regression).
- **Claim-level source anchors — the unsigned `claim_id` layer (#508; public #65 Q1).** A memory can now bind each claim it makes to the exact source span that justifies it: `palinode_save` (and REST `/save`, CLI `--claim TEXT::REF::QUOTE`, plugin) accept an optional `claims` list of `{claim_id?, text, source_id, span:{quote, quote_hash}, anchor_id?}` entries, persisted as plaintext frontmatter. `claim_id` is **content-addressed** — `sha256(memory_ref + normalized claim text)` truncated to 16 hex — derived on save when omitted and verified when supplied, so identical claim text in two memories gets two ids (no cross-memory aliasing) while identical claims within one memory dedup naturally; the claim text is stored alongside the id so the opaque `claim_id` stays resolvable to the claim it names. The `span` reuses the v0.9.0 `{quote, quote_hash}` source anchor **verbatim** (same normalization, same verifier): `quote_hash` stays the *integrity* half (does the cited text still match), `claim_id` adds the *addressing* half (a stable name for the claim→span binding) — composing with, not replacing, both file-level `fact:slug` identity and the `sources:` anchors (new optional frontmatter, absence-is-neutral, zero migration — the same discipline `sources`/`contradicts`/`backed_by` landed with). The read surface lands on blame, per the original public ask ("blame should answer *which source*, not just *when*"): `palinode_blame` with `claims=true`, `GET /blame/{path}?claims=true`, and `palinode blame --claims` resolve every claim to its cited span with live integrity status (`ok` / `anchor_tampered` / `source_drifted` / `source_missing`), a claim_id-still-matches-derivation check (catches renames/edits under a stored id), and whether the cited `source_id` is declared among the memory's `sources:` anchors. `palinode lint` gains an advisory `claim_anchor_issues` finding for bindings that no longer hold. The anchor shape is interop-aware toward yopedia's Phase-B anchor (`anchor_id` carried verbatim and nullable; `page` omitted for v1) pending coordination on public #65. Unsigned only — stable identifier + source-span pointer; no envelope or signing. Four-surface parity registered in `palinode/core/parity.py`; MCP schema golden fixture regenerated. New `palinode/core/claims.py` + `tests/test_claims_508.py` (42 tests: derivation, save round-trip + validation rejections, param-or-metadata resolution, blame resolution, lint, CLI flag) plus MCP schema/threading parity cases in `tests/test_mcp_parity.py`.
- **Sessions start smart — `palinode-session-start.sh` SessionStart hook for Claude Code (#261).** New `examples/hooks/palinode-session-start.sh` fires on Claude Code `SessionStart` (startup and `/clear` by default) and injects a bounded digest of your `core: true` memories into the fresh session as `additionalContext` — one `- [file] name — summary` line per memory (newest first, capped at 10 files / 4000 chars, tunable via `PALINODE_HOOK_INJECT_MAX_FILES` / `_MAX_CHARS`; `0` files = prime-only mode), prefixed with a deterministic reminder that recall goes through `palinode_search` / `palinode_read`. Standing context is present before the first prompt instead of depending on the agent remembering to search — the recall-fire phase of the memory loop is now deterministic on Claude Code, not instruction-hoped. The hook also POSTs `/context/prime` (forward-compatible server-side context warming: a harmless 404 until that endpoint ships, then it starts working with no re-install), supports token-protected deployments via `PALINODE_API_TOKEN`, and is fail-silent throughout — an unreachable API never blocks session start. `palinode init` now scaffolds it alongside the session-end hook and registers both in `.claude/settings.json` (idempotent; a pre-existing SessionEnd-only settings.json gains the SessionStart entry on re-run). New `tests/test_session_start_hook.py` (14 tests: prime POST, digest shape, source allowlist, fail-silent, bounds, bearer auth, and a byte-for-byte drift guard pinning the `palinode init` embedded copy to the canonical `examples/hooks/` script).

### Fixed

- **`palinode blame --claims` output no longer loses its bracketed labels to Rich markup (#262 follow-on to #508).** The claims-resolution text renders lines like `[ok] <claim_id>` and blame lines like `[git: 2026-…]`; Rich's `console.print` consumed the brackets as style tags, silently stripping the status labels from the terminal output. Both prints now pass `markup=False` (the new `palinode prime` command ships with the same guard).

## [0.9.2] — 2026-07-13

### Added

- **`docker compose up` — the one-command install (#613).** New `Dockerfile` + `docker-compose.yml` run the full stack in containers: API (loopback-only publish by default), watcher, bundled Ollama, and a one-shot `ollama-init` that pulls `bge-m3` (≈1.2 GB on first run, stated honestly — saves made before the pull finishes are keyword-searchable immediately and embed once the model is up). The memory dir is a **host bind mount** (`~/.palinode`, override with `PALINODE_DATA_DIR`) — files stay the source of truth and survive the containers. A host that already runs Ollama skips the bundled one: `OLLAMA_URL=http://host.docker.internal:11434 docker compose up palinode-api palinode-watcher` (the palinode services deliberately don't `depends_on` ollama, and `extra_hosts: host-gateway` makes the override work on Linux). The container entrypoint closes a first-run gap the bare install warns about: it `git init`s an empty data dir and sets a repo-local commit identity, so commit-on-save works out of the box. This doubles as the macOS/Windows service story — the README gains a **Running as a service** section (compose | systemd | launchd), including new `deploy/launchd/` LaunchAgent templates + README for venv-based macOS installs. The default deploy ships clean with **no Qdrant container** (resolves #376: the vector index lives in SQLite-vec inside `.palinode.db`; the Qdrant container in older internal deploys was a leftover no runtime path reads — the only Qdrant references in code are the one-time Mem0 *import*, which reads a user's existing Mem0/Qdrant as a migration source. A test now guards that Qdrant can't reappear in compose). New `tests/test_docker_compose.py` (13 checks across compose topology, entrypoint contract, Dockerfile, and launchd templates).

### Changed

- **README restructured for a cold reader — one ordered install path (#615).** The README had two competing getting-started sections — "Getting started in 60 seconds" (which presupposed Palinode was *already installed*) sat *above* the real first-timer "Quick Start", and **Requirements** (including the Ollama + bge-m3 dependency) sat at the very bottom, below Quick Start, Usage, Tools, Stack, and Configuration. A stranger hit the "already installed" path first and the requirements last. Restructured to the audit skeleton: **Requirements** (moved up, now with the "no Ollama → keyword-only mode" callout), then **Install** (one path: clone to `~/palinode-src` → venv → `pip install -e .` → point `PALINODE_DIR` at `~/.palinode` → `palinode doctor` as the "did it work?" gate, with the wrong-Python venv note inline), then **Connect your editor**, then **Daily use** (the demoted `palinode init` per-project scaffolding, retitled and moved below install). Pitch/architecture prose unchanged.

### Fixed

- **First save on a fully-cold host no longer blocks until the embed timeout (#619, completes #611).** Until an embed has succeeded in-process, `index_file` now runs one bounded `probe_embed` (~2 s, negative verdict cached 30 s) instead of letting every section pay the full embed timeout — the residual first-save stall #611 flagged as a follow-on. Deferred sections are written as **FTS-only rows**: chunks + BM25 index land immediately, no `chunks_vec` entry — which is exactly the missing-index signal the existing re-embed branch (#251 fix B) keys on, so rows converge to fully-embedded on the first pass after the embedder warms, with no new bookkeeping. This also makes the CLI's "saved and keyword-searchable now" note (#611) *true* — previously an embed failure skipped the chunk row entirely, so the memory wasn't BM25-findable until the watcher's retry succeeded — and it makes keyword-only installs (no Ollama at all) both fast (one cached probe per window, not a timeout per section) and immediately searchable. A proven-warm embed path skips the probe entirely — zero overhead in the healthy steady state. `store.upsert_chunks` treats an empty embedding as a deliberate FTS-only write (no vec attempt, no error log, `vec_ok` untouched). New `tests/test_cold_save_fast_return.py` (11 tests: gate logic + TTL cache, deferred FTS-only pass, warm-rerun convergence, no-downgrade of already-indexed rows, vector-less upsert, `has_embedded_ok` semantics) and a suite-wide conftest default pinning tests to the proven-warm path.
- **Cold/absent-Ollama first run no longer looks broken (#611).** On a fresh install where Ollama is up but `bge-m3` is unpulled or cold, Palinode reported healthy while nothing embedded — a confusing false-green. Four honesty fixes: (1) **`/status` gains `embed_functional`** — a real, bounded (~2 s), cached one-token embed probe (`OllamaClient.probe_embed`), distinct from `ollama_reachable` which only GETs the daemon and can't tell a cold model from a working one; on a cold host `ollama_reachable` is `true` but `embed_functional` is `false`, so monitors see the true keyword-only state. (2) **`palinode doctor` can no longer hang silently** — the diagnostics runner now enforces a per-check wall-clock budget (default 15 s) in a daemon thread, reports an over-budget check as a `warn` timeout and moves on, converts a check that *raises* into an `error` result instead of aborting the whole run, and supports streaming each result as it lands. (3) **First embed failure logs one plain line** — "embeddings unavailable → keyword-only (BM25) mode → `ollama pull bge-m3`", once per process, instead of only per-call WARNINGs. (4) **`palinode save` shows "embedding deferred"** — when the inline embed doesn't complete, the CLI now prints a yellow "saved and keyword-searchable now; semantic recall follows" note under the green Saved line instead of leaving the impression the save failed (the file is always persisted and git-committed before embedding). New `tests/test_first_run_ux.py`. *(Note: a fully-cold first save can still block until the embed timeout before the circuit breaker opens; making that path fast-return unconditionally is a follow-on.)*
- **Per-harness MCP install docs point the Claude Code CLI at the wrong config file (#612).** `docs/INSTALL-CLAUDE-CODE.md` told users to hand-edit `~/.claude/claude_desktop_config.json` to wire the Claude Code **CLI** — but that file belongs to the Claude **Desktop** app; the CLI reads `~/.claude.json` / project `.mcp.json`. The stdio and remote-HTTP sections and the troubleshooting step now use the canonical CLI commands (`claude mcp add palinode -- palinode-mcp`, `claude mcp add --transport http palinode <url>`, `claude mcp list`, `palinode mcp-config --stdio/--http/--diagnose`) instead of hand-printed JSON that drifts, and the README gains a single **Connect your editor** table that delegates to the CLI per harness. Deep per-harness detail stays in `docs/MCP-INSTALL-RECIPES.md`.
- **Documentation drift sweep — tool counts, versions, sizes, dead install ads (#614).** A newcomer's first ten minutes hit several trust-eroding inconsistencies, now reconciled to ground truth: the MCP tool count (stated as **25 / 21 / 17** across README, `INSTALL-CLAUDE-CODE.md`, and the plugin README) is unified to the real **27**; the plugin README's stale `v0.6.1` / "26 commands" become the current CLI (40+ commands) with a generic CHANGELOG link; "PyPI publish coming soon" is replaced with the real `pip install palinode` / `uvx palinode` (0.9.1 is on PyPI); the BGE-M3 download size is corrected from a contradictory `568MB`/`1.2 GB` pair to `≈1.2 GB`; the stray `Qdrant (Mem0, separate) 6333` row — an internal-system reference that doesn't belong to Palinode — is removed from the QUICKSTART ports table; and the plugin-marketplace handle is corrected to the public org (`palinode@phasespace-labs`). (Deferred to the README restructure: the single-canonical-tool-table refactor so other docs *link* rather than restate, the `palinode --version` log-noise quieting, and the `init_db` first-run one-truth harmonization.)
- **Memory-dir split-brain in the install docs + shipped config example (#610).** The README Quick Start exports `PALINODE_DIR=~/.palinode`, but the shipped `palinode.config.yaml.example` set `memory_dir: "~/palinode"` (no dot) — copy the config as instructed and the two fight: `PALINODE_DIR` wins for some paths while the stale `memory_dir` leaks into others, so files write to one directory while the DB attaches to another and search silently returns nothing (the README's own Diagnose section names this as the most common failure). The config example now **comments out `memory_dir`** so a copied config inherits `PALINODE_DIR` instead of overriding it, and the docs are unified on one convention: memory lives at `~/.palinode` (via `PALINODE_DIR`), code clones to `~/palinode-src` (never the memory dir — `docs/INSTALL-CLAUDE-CODE.md` previously cloned into `~/palinode`, colliding with where the plugin docs put memory). `docs/DEPLOYMENT-GUIDE.md` and `claude-plugin/README.md` no longer use the dotless `~/palinode` for the store. The `palinode doctor` split-brain guards this already covered (`multiple_palinode_dirs`, `db_path_under_memory_dir`, `config_drift`, `phantom_db`) are unchanged — this closes the docs/config half that was *creating* the condition.

## [0.9.1] — 2026-07-04

### Added

- **`palinode worktree-reconcile` — reclaim stale dead-PID-locked git worktrees (#448).** Claude Code (and similar agents) create isolated worktrees under `.claude/worktrees/` locked with the owning session's PID; when a session crashes the lock outlives the process, so the worktree is never reclaimed — it bloats the repo and pollutes file-watch/index globs (the Palinode watcher sees ghost branches). The new command reconciles them **safely**: a locked worktree is removed only when its lock PID is dead (`os.kill` → `ProcessLookupError`) **and** its tree is clean **and** its branch has an upstream (so nothing unpushed is lost — `git worktree remove` drops only the directory; branch + commits survive). Anything alive, dirty, or lacking an upstream is skipped with a stated reason. **Dry-run by default**; `--execute` unlocks + force-removes + prunes; `--json` for scripting. Additionally, `palinode init` now scaffolds three `permissions.allow` rules (`Bash(git worktree remove/unlock/prune:*)`) into `.claude/settings.json` so agents can run the cleanup without tripping the auto-mode permission classifier (merged idempotently on re-run). New `tests/test_worktree_reconcile.py` (real git + bare-remote fixtures across the dead/alive/dirty/no-upstream cases and the CLI dry-run/execute paths) and init allow-rule tests.
- **`palinode mcp-config` now warns *live* when Claude Desktop is running (#373).** Editing `claude_desktop_config.json` while the app is open loses the edit — Claude Desktop rewrites that file from memory on quit (and silently strips `url`-form entries). The diagnostic already carried an always-on static "quit first" note; it now also does a **live process check** (`claude_desktop_running()`, previously only wired into `mcp-smoke`) and escalates to a loud "⚠ Claude Desktop appears to be RUNNING right now" warning when the app is actually up — while a `None` ("couldn't tell") verdict stays quiet rather than crying wolf. This completes the running-process auto-detection deferred from the v0.8.13 #373 fix, wired into the command users actually run to configure Claude Desktop. The pipe-friendly `--http`/`--stdio` emit modes stay clean (warning is diagnose-only). New `TestClaudeDesktopLiveWarning` in `tests/test_mcp_config_diagnose.py`.
- **`llms-install.md` — agent-bootstrap install recipe (#271).** A repo-root file that coding agents (Cline, Cursor, Claude Code, …) read preferentially over the README when installing autonomously. Unlike the narrative README, it is a top-to-bottom runnable recipe with **prerequisite detection built in**: shell one-liners that print `OK`/`MISSING` for the Ollama daemon and the `bge-m3` model, each followed by the macOS/Linux install-and-branch step — so a clean machine without Ollama gets it configured instead of failing. Every service-start and MCP command uses the **absolute venv path** (`$VENV/bin/…`) to avoid the wrong-Python trap (a venv script resolving its deps against system Python, e.g. a spurious `httpcore missing`), and Step 5 has the agent substitute that absolute path into the `palinode mcp-config --stdio` block. Ends with a `palinode doctor` + `palinode_status` verification gate before declaring success, plus a troubleshooting table. Motivated by a Cline preflight that self-recovered from both friction points but shouldn't have had to.

### Fixed

- **Oversized streamed (chunked) request bodies now return 413, not 400 — and it's actually tested (#297).** The streaming body-size guard (`_BodySizeLimitMiddleware`, #286) aborts an over-limit body by raising from the wrapped ASGI `receive`, but Starlette's body parser caught that exception first and surfaced its own **400 "error parsing the body"** — so a chunked upload over the limit was rejected (good) but with the wrong, misleading status (the intended **413 Payload Too Large** only ever fired for a raw ASGI caller). The middleware now wraps `send` too: once the limit trips, it overrides the framework's error response with the correct 413 end-to-end. Surfaced by closing the test gap the 2026-05-01 smoke run flagged — `curl --data-binary @file` sets `Content-Length` and takes the header fast-path, so the streaming byte-counting path was never exercised. New tests in `tests/test_body_size_limit.py`: a deterministic middleware-level check that the oversized tail is **never read** once the limit is crossed (413 fires mid-stream), and an end-to-end **real uvicorn server + raw-socket `Transfer-Encoding: chunked`** upload asserting a genuine 413 (the shape the smoke test intended but couldn't send).
- **FTS5 external-content `chunks_fts` no longer orphans rows after delete/supersede (#439).** `chunks_fts` is declared `content=chunks`; the chunk-delete paths removed its rows with a bare `DELETE … WHERE rowid=…`, which on an external-content FTS5 table can leave the inverted-index tokens behind (`count(chunks_fts) > count(chunks)`), so BM25 keyword recall was silently wrong — "I saved it but it doesn't show up in search" — until an unrelated save happened to trigger `rebuild_fts()`. All three delete paths (`store.delete_file_chunks`, the `index_file` re-index prune, and the `upsert_chunks` FTS retry branch) now route through a single `store.fts5_delete_chunk()` helper that issues the sanctioned FTS5 `'delete'` command (re-deriving the row's tokens from its still-present column values), keeping `count(chunks) == count(chunks_fts)` without any rebuild. The `upsert_chunks` "never DELETE-first against an empty FTS5 table" guard is preserved. New `tests/test_fts5_lockstep_delete.py` asserts lockstep across both production delete paths with no `rebuild_fts()` call, plus a build-independent guard that the bare idiom can't return.
- **MCP HTTP transport now warns on a token-less `0.0.0.0` bind — parity with the API server (#253).** The streamable-HTTP MCP transport (`palinode-mcp` serving `/mcp/`, #217) defaults to binding `0.0.0.0`, which exposes the full tool surface (`palinode_save`/`palinode_search`/`palinode_read`/…) on every interface. ## [0.9.0] — 2026-06-28

### Added

- **Periodic project-memory review — `palinode review [project]` (#366).** A first-class "step back and review the whole project's memory" pass. It **composes** the deterministic health signals (stale active files, long-unresolved `open_question`s #72, open `contradicts` links #533, orphans, missing descriptions, wiki drift) scoped to a project, and translates the findings into **proposed** corrective ops in the executor's vocabulary (`PROPOSE_ARCHIVE`/`PROPOSE_UPDATE`/`PROPOSE_SUPERSEDE`) — **advisory and read-only: it proposes, never applies** (mirroring #533's `PROPOSE_CONTRADICTS` discipline; `SUPERSEDE` stays the only winner-picking op). Project scoping reads the `project/<slug>` entity from frontmatter directly, so the pass is **deterministic and offline** (no embedder/DB dependency) and safe to run on a nightly/weekly cron; embedding-based near-duplicate detection stays the explicit `dedup-suggest` call (surfaced as a hint in the report). Lands on CLI (`palinode review [project] [--format json|text]`), MCP (`palinode_review`), and REST (`POST /review`) per ADR-010 parity (plugin-exempt like `lint`/`topic_coverage`/`consolidate`), registered in `palinode/core/parity.py`. New `palinode/core/review.py` (`run_review`) + `tests/test_review.py` (whole-store findings, project scoping incl. typed refs and empty scope, read-only guarantee, proposed-op mapping, API endpoint); MCP schema golden and API route-inventory fixtures regenerated for the new tool/route.

- **`palinode mcp-config` emits ready-to-paste transport config blocks — stdio → streamable-HTTP migration (#217).** The streamable-HTTP MCP endpoint (`palinode-mcp` serving `/mcp/` on port 6341) is already deployed; the remaining migration work is client-config emission + docs. `palinode mcp-config` gains `--http` and `--stdio` emit modes alongside the existing default diagnostic: `--http` prints a copy-pasteable `{ "type": "http", "url": "http://<palinode-host>:6341/mcp/" }` block (with `--host` / `--port` / `--url` overrides), `--stdio` prints the local `command: palinode-mcp` block. TTY-aware — piped output is the raw JSON block (redirect straight into a config file), interactive output adds guidance and a `claude mcp add` one-liner. For forward-compat with bearer-auth parity (#289), `--http --bearer <token>` adds an `Authorization` header. The command stays read-only (never writes a user config file). `docs/MCP-CONFIG-HOMES.md` documents the stdio-vs-streamable-HTTP transport forms, the migration rationale (warm BGE-M3 model, no per-session Python cold-start, no persistent SSH tunnel), and the Claude-Desktop-only-accepts-stdio caveat. New `TestEmitBuilders` / `TestEmitCommand` in `tests/test_mcp_config_diagnose.py` cover URL resolution, both block shapes, the bearer header, and the `--http`+`--stdio` conflict.
- **Mechanical untyped cross-linking — `cross_refs` (#73).** During indexing the watcher now scans a memory's body for deterministic mentions of *other* memory files and records the matches in an untyped `cross_refs` frontmatter list — the machine-generated complement to the authored `entities:` ↔ `[[wikilinks]]` pair (PROGRAM.md wiki-maintenance contract). Matching is conservative and whole-token/whole-phrase (case-insensitive): a candidate is matched by its full `category/slug` path ref, its slug (only when hyphenated or ≥ `min_token_len`), or its distinctive title (multi-word or ≥ `min_token_len`, excluding generic stopword titles), so short/generic names like `api` or a memory titled `Status` never produce false links — "untyped refs are always correct". Refs are directional (only the scanned file's outbound mentions; no back-references), self-references are excluded, and the pass is **idempotent**: the file is rewritten + committed through the mutation choke point (`git_tools.write_memory_file` / `commit_memory_file`) only when the computed `cross_refs` actually change, so the watcher re-processing its own write terminates after one pass (no write-amplification loop). Gated by `capture.cross_refs.enabled` (default on) with a `capture.cross_refs.min_token_len` floor (default 6). The new `palinode/core/cross_refs.py` holds the pure matchers (`build_registry` / `detect_refs`) and the file-mutating `update_file_cross_refs`; the watcher (`palinode/indexer/watcher.py`) calls it as a post-index hook. **Scope deconfliction:** #73 delivers only the mechanical *untyped* layer — typed relations (`contradicts` / `backed_by`) are owned by #533, which builds on this. New `tests/test_cross_refs.py` covers the matchers (full-ref / slug / title hits, short-slug + stopword-title skips, whole-token boundaries, sorting), registry self/skip-dir exclusion, and the orchestrator (write-on-mention, idempotency, stale-ref removal, disabled-config no-op, self-exclusion).
- **Epistemic markers — a memory declares what KIND of claim it makes (#72, ADR-018).** A new optional frontmatter field `epistemic: fact | inference | open_question` lets a memory state its epistemic weight, orthogonal to `type` and `status`: `fact` (directly observed/verified), `inference` (derived/extrapolated — lower trust; should carry a `backed_by` once #533 lands), or `open_question` (explicitly unresolved). Absence of the field is its own trust-neutral state — `unmarked` (no epistemic claim made), deliberately **not** equated with `fact`: for an audit-grade store, "nobody declared this" must not inherit the authority of "verified". `unmarked` is neither flagged by lint nor asserted as verified by downstream consumers; `fact` is an explicit, earned marker. Every memory predating the field is `unmarked` and byte-for-byte unaffected; the field is persisted only when a marker is set. Settable on save across **all four surfaces** (ADR-010 parity) — MCP `palinode_save` enum param, REST `SaveRequest.epistemic`, CLI `palinode save --epistemic`, and the OpenClaw plugin — with the vocabulary as a single source of truth in `palinode/core/parser.py` (`VALID_EPISTEMICS` / `DEFAULT_EPISTEMIC`). `unmarked` is not a settable value (reached only by omission); an unknown value is rejected at the `/save` surface with HTTP 400 whether it arrives via the first-class param or the free-form `metadata` dict (the explicit param wins, validated once). On recall, asserted markers are labelled in MCP search results and the `/ui` provenance panel renders a **Claim type** row (`unmarked` trust-neutral, `open_question` in warn styling); `palinode lint` reports `stale_open_questions` — an `open_question` older than the 90-day stale threshold wants resolution. The marker is **sticky** (like `update_policy`/`created_at`): it carries forward across a re-save of the same slug that omits it, so a `fact`/`inference`/`open_question` is never silently dropped back to `unmarked` — to change it the writer states the new value explicitly. Foundation for the typed contradiction/evidence-backing links (#533) and the advisory project-memory review (#366). New `tests/test_epistemic_markers.py` (round-trip, default-not-written, invalid-rejected via param and metadata, CLI threading, lint staleness) plus MCP schema + forwarding parity cases in `tests/test_mcp_parity.py`.
- **Typed contradiction + evidence-backing links — `contradicts` / `backed_by` (#533, G4).** Supersession already expresses *resolved* disagreement (one fact wins, the loser is struck through and logged to history); audit-grade provenance also needs the two relationships it can't represent. `palinode save` now accepts two first-class **plaintext** frontmatter lists of `category/slug` refs: `contradicts: [...]` ("these conflict, neither wins yet — surfaced for review") and `backed_by: [...]` ("this fact is supported by that source/fact"). Both land on all four surfaces per ADR-010 — MCP (`palinode_save` schema + body-forward), REST (`SaveRequest` on `POST /save`), CLI (repeatable `--contradicts` / `--backed-by`), plugin (`palinode_save` TypeBox schema) — and are registered as canonical params in `palinode/core/parity.py`. Malformed refs (traversal, absolute paths, empty/non-string) are rejected at the save surface with HTTP 400, resolved from param-or-`metadata` (param wins) so a tunneled ref is validated too. `contradicts` is **bidirectional**: declaring `contradicts: [B]` adds a best-effort reciprocal back-link into B's own list (idempotent, via the `git_tools` write/commit choke point; never raises, never blocks the save — a missing target is skipped). `palinode lint` reports unresolved conflicts as a new `open_contradictions` finding (any file carrying a non-empty `contradicts` list) on every surface. The consolidation executor gains a non-destructive `PROPOSE_CONTRADICTS` op that records a contradicts link **without picking a winner** — `SUPERSEDE` stays the only winner-picking operation, and the op is exempt from the `update_policy: replace` guard since recording a conflict forks nothing into history. Ships PLAINTEXT only (the signed attestation layer is out of scope per ADR-016 §4). Independent of #73's untyped `cross_refs` (PR #590, not yet in main) — `contradicts`/`backed_by` are their own keys, the typed layer over #73's mechanical layer. New `tests/test_typed_links.py` covers persist/round-trip on both keys, the seeded conflicting pair (reciprocal back-link), 400 on malformed refs, the lint finding, and executor propose-not-resolve.
- **Cross-surface parity inventory guard — surface→registry drift (#170).** The ADR-010 parity test already walked the registry and checked each surface (registry→surface), but that direction is blind to a *new* capability shipped on a surface and never registered — the test only iterates operations it already knows about. `tests/test_surface_parity.py` now enumerates the **live** capabilities of each surface (MCP `list_tools()`, the FastAPI `app.routes`, the Click command tree) and asserts every one is accounted for by exactly one of three buckets in `palinode/core/parity.py`: `REGISTRY` (a parity-bound memory op), `INVENTORY_INFRA` (framework/admin/observability surface — Swagger/Redoc/OpenAPI, the `/ui` inspector, liveness probes, DB-maintenance + importer endpoints; the surface-identifier form of `ADMIN_EXEMPT_OPERATIONS`), or `INVENTORY_BACKLOG` (a memory-semantic op that ships today but isn't yet promoted into the registry, each mapped to its tracking issue — the #170 implementation backlog). The MCP `palinode_status` tool renders a `Version:` line and `palinode_doctor` passes the field through; `palinode status` (text view) and `palinode doctor` (header) mirror it, honoring the all-four-interfaces parity convention. Four behavior changes: **(1) explicit-only gate** — only `mode == "explicit"` recalls nudge `importance`; passive/ambient/session-start priming (ADR-008/#358) does not (the memory was *offered*, not *sought*). `recall_count`/`last_recalled` still update on every hit (raw frequency/recency). `mode`/`session_id` now thread from `/search` (`SearchRequest`) → `store.search`/`store.search_hybrid` → `store.record_recall`; existing callers default to `explicit`, so behavior is unchanged until an ambient caller sets `mode="passive"` (the gate is built load-bearing for that path — see the STEP-0 finding in #86). **(2) session-deduplication** — `importance` is nudged at most once per `(chunk, session_id)`, recorded in a new `recall_nudges` table (`INSERT OR IGNORE` membership test); two hits on the same memory in one session = one reinforcement, but `recall_count` stays raw. **(3) exponential-approach reinforcement** — replaces the additive `+0.01` nudge with `importance ← importance + (cap − importance)·α` (α = 0.08), smooth and self-limiting; a median-2-demand memory lands ≈0.57, a p90-26-demand memory ≈0.90, saturating at the cap. **(4) decay-on-read at rank time** — when the decay ranker is enabled, ordering uses an *effective* importance `eff = base + (importance − base)·exp(−Δt/τ)` (τ = 14 d), **floored at base** so cold memories are never demoted below neutral; `eff` enters as a *bounded* re-rank boost (band 0.25, boost ∈ [1.0, 1.25]) that modulates within the relevance ranking rather than leading it — cold/unrecalled memories remain findable. New `DecayConfig` constants `importance_base`/`importance_cap`/`importance_alpha`/`importance_tau_days`; the per-type taus (`tau_critical` … `tau_ephemeral`) and the old additive `importance_nudge` are **deprecated** (retained for config-load compatibility, no longer the decay clock — see §3.4). `palinode doctor`'s `recall_write_health` check now also reports importance spread (are hot memories separating from base, or stuck at 0.5 — a flat distribution after sustained explicit cross-session demand hints the mode gate is mislabeled). New `tests/test_demand_decay_importance.py` (real SQLite + tmp_path) covers the explicit/passive gate, session dedup, the exponential-approach milestones at n∈{1,2,26}, decay-on-read halfway/floor behavior, and the cold-memory findability regression.

- **Consolidation never SUPERSEDEs/ARCHIVEs a living (`update_policy: replace`) document (ADR-015 §2.2, #431 §3).** The deterministic executor (`apply_operations`) now reads the target file's declared write-semantics regime once and, when it is `update_policy: replace`, **refuses** any `SUPERSEDE` or `ARCHIVE` op against it — these are the two ops that fork content into history (strikethrough + `supersedes-` sibling, or removal into the `-history.md` file), which would split the one current fact into a stale snapshot, the exact failure the axis exists to prevent. Refused ops are logged and counted under a new `protected_rejected` stat (flows through the runner/write-time stat aggregation as an opaque key). `UPDATE`/`MERGE`/`KEEP`/`RETRACT` remain allowed — a living doc is *updated in place*, never archived. A normal episodic doc (no `update_policy`, or `append`) supersedes/archives exactly as before — the guard is a strict no-op for it. Parsing the regime falls open to "no protection" on a malformed file so the guard can never block consolidation. New `tests/test_executor_replace_guard.py`.

- **Writer-supplied `status` is now validated against a combined lifecycle + incident allow-set (ADR-015 §2.2, #398).** `save_api` validates an incoming `status` (via `metadata.status`) against `VALID_STATUSES` = the existing KU lifecycle vocabulary (`active`/`archived`/`deprecated`) **plus** the incident lifecycle (`open`/`monitoring`/`resolved`); an unknown value is rejected (HTTP 400). The incident values are deliberately **disjoint** from the lifecycle values and from the default search-exclusion set (`config.search.exclude_status == ["archived"]`), so incident lifecycle and status-based recall exclusion share the one `status` field **without colliding**: an `open`/`monitoring`/`resolved` incident is never silently excluded from recall, while `archived` is still excluded exactly as before. This is the deliberate reconciliation choice — keep lifecycle in the existing `status` field rather than minting a separate field, because the vocabularies don't overlap (covered by the parametrized status tests + the two exclusion-regression tests in `tests/test_update_policy_axis.py`).

A file is eligible iff it lives directly under a memory-category directory (`people`/`decisions`/`projects`/`insights`/`research`/`inbox`); structural and top-level files are excluded. On a tree of only structural files, `pending_descriptions` is now `0`. New `TestDescriptionEligibility` in `tests/test_auto_summary_async.py` (real files + `tmp_path`) covers the predicate, count-exclusion, worklist-exclusion (generator never invoked), and an eligible-file regression guard. No write-back behavior changed.
- **`/read` recall-recording now uses the resolved absolute path (#479).** `store.record_recall_for_paths` matched on the caller-supplied relative path (`"insights/foo.md"`), but `index_file` stores chunks with the absolute path (`/var/lib/palinode/insights/foo.md`). The WHERE clause never matched any rows, so every `/read` call silently left `recall_count=0` and `last_recalled=NULL` — the exact integration seam #371's store-level tests passed green over. Fix: pass `resolved` (the absolute path already computed by `_resolve_memory_path`) instead of the relative `file_path` to `record_recall_for_paths`. New `tests/test_read_recall_e2e.py` covers the full save → index → `/read` → DB-assert pipeline end-to-end.

- **Replace-guard fail-open now emits a WARNING; `search()` / `search_fts()` connection hygiene (#483).** Two minor robustness fixes: (1) `_is_replace_policy` in the consolidation executor previously fell open silently when frontmatter was unreadable — a corrupted living document lost its SUPERSEDE/ARCHIVE/RETRACT protection with no observable signal. Now logs a `WARNING` on the fail-open path; when the raw text contains `update_policy: replace` the warning includes an explicit "may be unprotected" hint. Fail-open behaviour is unchanged (consolidation is never blocked). (2) `store.search()` and `store.search_fts()` called `db.close()` only on the happy path — an exception during row processing left the connection open. Both now use `try/finally` to guarantee close on all paths. Existing test suite passes; new `tests/test_robustness_483.py`.

- **Priority ranking weight `_PRIORITY_RANK_WEIGHT` is now test-tuned, not hard-coded-pending (#486, follow-up to #259/#478).** #259 specified the human-priority RRF nudge should be tuned against a ranking test; it shipped hard-coded at `0.025` with a "pending empirical tuning" note. New `tests/test_priority_ranking_486.py` pins the two ordering properties that justify the value — (1) a strong match at normal priority (3) still outranks a clearly weaker match at max priority (5), and (2) similar-relevance hits are ordered by priority — by mocking the RRF inputs so the expected ordering is deterministic (no magic fused score is asserted). A teeth test proves the weight is load-bearing: inflating it lets a max-priority weak match overtake a stronger one. `0.025` (a ±0.05 band over priority 1..5) is confirmed as the tuned value with the test as its rationale; the `store.py` comment is updated accordingly.

- **ADR-015 write-semantics hardening — four post-merge defects found by adversarial review (H1–H4).** A multi-dimension review of the merged write-semantics work surfaced four production-reachable defects the existing suite passed green over:
  - **H1 — internal reads no longer pollute recall stats.** `store.search()` records access metadata by default (correct for human recall), but two *internal* callers omitted the opt-out, so every consolidation dedup lookup (`consolidation/runner.py`) and every dedup/orphan/cluster/topic-coverage candidate scan (`api/server.py::_embedding_candidates`) bumped `recall_count` and nudged `importance` on over-fetched slates nobody recalled — re-introducing, from a new path, exactly the monitoring-churn pollution the #398 telemetry-exclusion prevents. Both now pass `record_access=False`, matching the `search_hybrid` inner-pass pattern. (Chose the surgical opt-out over inverting the default, which would have broken the documented `search()`-records-by-default contract in `test_recall_feedback.py`.)
  - **H2 — re-index no longer wipes accumulated recall metadata.** `upsert_chunks` used `INSERT OR REPLACE`, whose column list omits `importance`/`recall_count`/`last_recalled`; SQLite's delete-and-reinsert reverted them to schema defaults on *every* content change (consolidation UPDATE/MERGE, manual edit, sticky-frontmatter rewrite). With #86 making the ranker read those columns, the reinforcement signal was being destroyed precisely when a memory was actively used. Switched to `INSERT … ON CONFLICT(id) DO UPDATE` that refreshes only the content columns and leaves the recall signal intact.
  - **H3 — RETRACT no longer forks a living (`update_policy: replace`) document.** The §2.2 guard refused SUPERSEDE/ARCHIVE on a replace-doc but allowed RETRACT — yet `_retract_fact` strikethrough-tombstones the current fact in place *and* appends a `-history.md` sibling, the same history-fork the guard exists to prevent (RETRACT is LLM-emitted for provably-wrong facts per the compaction prompt). RETRACT is now guarded alongside SUPERSEDE/ARCHIVE on replace-docs; a wrong value in a living doc is corrected with UPDATE. Still allowed on normal episodic docs.
  - **H4 — `update_policy` supplied via the `metadata` dict is now validated.** Only the first-class `update_policy` param was validated; a value passed through `metadata={...}` skipped validation, merged into frontmatter verbatim, and could silently arm the executor replace-guard. `save_api` now resolves the effective policy from param-or-metadata (param wins), validates that (HTTP 400 on unknown), and writes the single validated value — mirroring how `status` was already handled.
  - Regression tests added for all four: internal-read non-pollution + re-index durability (`tests/test_recall_feedback.py`), RETRACT-on-replace rejection + episodic parity (`tests/test_executor_replace_guard.py`), metadata-bypass validation + param-precedence (`tests/test_update_policy_axis.py`).

- **`ARCHIVE` no longer leaves obsolete facts in default recall (#485).** The deterministic executor's `ARCHIVE` op (and the old-version side of `SUPERSEDE`) relocates a fact into a `{base}-history.md` sibling — but that history file was created with no `status` frontmatter, so the indexer chunked it as `active` and `config.search.exclude_status = ["archived"]` never suppressed it. The one class of fact `ARCHIVE` exists to hide kept resurfacing in default `/search`, just sourced from the history file — contradicting PROGRAM.md's "move to `status: archived`, never hard-delete" contract. Fix: history files now carry `status: archived` frontmatter (new-file template + a `_ensure_archived_frontmatter` injector that upgrades legacy history files on the next append), so archived/superseded content is excluded from default recall while staying indexed and retrievable on demand (`status_exclude_list=[]`). PROGRAM.md documents the relocate-to-history mechanism explicitly. New `tests/test_archive_recall_485.py` proves the archived fact is absent from default recall and present with the explicit-include path (real SQLite + tmp_path), and the `test_executor.py` ARCHIVE test now asserts the resolved contract.

- **`/wrap` ships its own session note via `session_end(push=true)`, not a forgettable trailing push (#378, refines #353).** The `palinode init` `/wrap` scaffold (`WRAP_COMMAND_BODY`), the heavy-wrap scaffold (`WRAP_HEAVY_COMMAND_BODY`), and the dev command called `palinode_push` before `palinode_session_end`. But `session_end` *commits* the daily note / status / indexed file and only *pushes* when `config.git.auto_push` is enabled (default off) — so the push-before-only flow left the wrap's own session note unpushed until some later push, and the final session before a gap never reached the remote. The fix folds the note-ship **into the tool** via the new `push` param (see Added): the flow is now `palinode_push` (sync prior work, #353) → `palinode_session_end` **with `push: true`** (commit *and* ship the note in one call). This is strictly more robust than the earlier "push-twice prose" intermediate — even an agent that botches the sequence and only calls `session_end(push=true)` still ships the note, because the guarantee no longer depends on remembering a third step. The heavy wrap had the same latent bug (its Step 4 note, committed *after* the Step 2 push, was never pushed) and is fixed the same way. `tests/test_wrap_push.py` now guards that both scaffolds invoke `session_end` with `push: true`.

- **`created_at` is now preserved on existing-slug overwrite (ADR-015 §2.4, #430).** Previously `save_api()` re-stamped *both* `created_at` and `last_updated` to now on every write, including overwrites of an existing slug — so re-saving the same logical memory destroyed its first-seen timestamp, the engine defect behind the #430 living-document churn (144 `uptime-kuma` files in ~1.5 days, each a distinct timestamped slug). Fix: before building the frontmatter, if the target file already exists, parse its existing `created_at` and carry it forward; only `last_updated` advances to now. A genuinely new slug still stamps `created_at = now`. This is deliberately **not** gated behind `update_policy` (that param is PR-B) — re-saving the same `(category, slug)` is the same logical memory, so birth-time preservation is the correct unconditional default. Fallback when an existing file lacks `created_at` in frontmatter: stamp now (the git-log first-commit lookup in ADR-015 §2.4 is deferred to a later refinement). New `tests/test_created_at_preservation.py` (real SQLite + tmp_path) covers overwrite preservation across N updates, brand-new-slug stamping, per-slug independence, and the no-`created_at` fallback.

- **Recall-feedback loop now actually writes access metadata back to `chunks` (ADR-006/007, #371).** The retrieval hook fired (the audit log proved ~2,049 retrievals/7 days) but `recall_count` stayed `0`, `last_recalled` stayed `NULL`, and `importance` stayed `0.5` across the whole corpus — so every downstream recency/importance/decay policy ran on null data. Root cause: `store.search()` and `store.search_fts()` omitted the chunk `id` from their result dicts, so the write-back already present in `search_hybrid` (`r.get("id")`) silently no-oped. Fix: surface `id` in both result dicts and centralize the write-back in the shared store layer — a single batched, resilient `record_recall()` (and `record_recall_for_paths()` for the read path) increments `recall_count`, stamps `last_recalled` (tz-aware UTC ISO-8601), and nudges `importance` upward by `decay.importance_nudge` toward `decay.importance_cap` (bounded reinforcement — ADR-007 leaves the importance-update rule unspecified; this is the simplest faithful reading). Wired into `/search` (vector + hybrid) and `/read`; MCP and CLI inherit it through the API (ADR-010 parity). Write failures are logged and swallowed — the latency-sensitive read path never fails on a stats write. Access-metadata writes happen regardless of the `decay.enabled` ranker flag (that flag only gates the `score_with_decay` re-ranking term). New `tests/test_recall_feedback.py` covers stamping exactly the hit chunks, accumulation across repeated recall, the importance nudge + cap, write-failure resilience, the file-path (read) variant, and the doctor health check — all on real SQLite + tmp_path.

- **Heavy `/wrap` template is now non-GitHub aware (#440, follow-up to #419).** Found by dogfooding the heavy `/wrap` against a non-GitHub **Gitea** repo. Three GitHub-first assumptions in `WRAP_HEAVY_COMMAND_BODY`: (1) the **merge** step ran `gh pr list` and a "no known GitHub host" error could read as a *halt* rather than a graceful skip — now it explicitly skips merge (not a halt) on a non-GitHub remote and points at the host's own CLI (`tea` for Gitea); (2) the **push** step pushed all unpushed commits blind — now it states the all-or-nothing assumption, **lists** `@{u}..HEAD` first, and treats a committed-but-not-ready commit as a stop-and-ask; (3) the **triage** auto-file sub-step assumed `gh` — now names the per-host tool. Template + tests only (`palinode/cli/init.py`, `tests/test_init_wrap_heavy.py`); no runtime code path.

This clears the last direct-`httpx`-to-Ollama calls: all Ollama I/O in `palinode/` now goes through `OllamaClient` (#338 acceptance criterion met).

- **Centralized `palinode/core/ollama_client.py` — single mediation layer for all Ollama I/O (#338, Phase 1+2+3 foundation).** Before this, every caller (embedder, the `/api/generate` description/summary paths, consolidation, lint) built its own `httpx` request, set its own timeout, and swallowed its own errors — palinode was an unmediated dependency on a single Ollama host's instantaneous responsiveness, which is the architecture behind the recurring #335/#336-class incidents. The new `OllamaClient` provides: **per-role routing** (`OllamaRole.EMBED` → embed host, `CHAT` → chat/generate host, `CONSOLIDATION` → its own URL), with typed methods (`embed`/`generate`/`chat`/`show`) bound to a role so an embed call cannot be sent to the chat host by construction; **retry with jittered exponential backoff** on transient failures (read/connect timeout, 5xx), with per-call `retries=0` to opt latency-sensitive paths out; **a per-role circuit breaker that opens loudly** (WARNING + surfaced state) and fast-fails subsequent calls with a typed `OllamaCircuitOpen` in sub-millisecond time instead of waiting for a timeout, half-opening after a cooldown to probe recovery; **typed exceptions** (`OllamaError`/`OllamaUnreachable`/`OllamaTimeout`/`OllamaCircuitOpen`) so callers never receive a bare httpx error; **structured JSON-line logging** (#337 convention: `{event, op, role, endpoint, model, latency_ms, retry_count, circuit_state, outcome}`); and **rolling per-role latency/error metrics** (`p50_ms`/`p95_ms`/`error_rate_5m`, 5-minute window) for a future `/status` + `palinode doctor` surface. Additive only — no existing caller is routed through it yet; callers migrate in follow-on phases (embed path, generate paths, consolidation) under #338, each needing smoke-rig validation. 22-test unit suite in `tests/test_ollama_client.py` drives the acceptance criteria (typed errors, breaker opens at threshold, open-circuit fast-fail does zero network I/O, per-role routing, structured fields, percentile metrics).

### Changed

- **Auto-description is now fully async — `/save` never blocks on the LLM (#405, follow-up to #336/#403).** The inline `_generate_description` call was the last LLM hop on the `/save` hot path: #336 capped it with a timeout/circuit-breaker, but a *warm-but-contended* chat model still blocked the response for up to `describe_timeout_seconds` (default 5 s), surfacing to sub-10 s REST clients as a "write timeout" while the durable write had already committed. `/save` now defers the description exactly like auto_summary (#403) — it writes the file without a `description` field, returns `description_pending: true` for eligible saves, and returns in `embed_time + write_time` (typically < 500 ms) regardless of model state. `config.auto_summary.enabled` is now the single master switch for *all* LLM enrichment (description **and** summary): disabling it makes `/save` fast unconditionally and stops both description and summary scheduling. A caller-supplied `description` (via `metadata`) is still respected and never deferred. The watcher-driven `/generate-summaries` backfill lands the description out of band (~30 s); the file is born with no `description` key during the window, by design. **Also fixes a latent bug from #336:** the watcher's description-retry called `POST /generate-summaries`, but that endpoint only filled *summaries* — deferred descriptions were never actually backfilled. `/generate-summaries` now fills both (descriptions are not core-gated; every file missing one gets it), and exposes `descriptions_generated` / `description_errors` in its response. Observability: `/status.auto_summary` and `/health/auto-summary` gained `last_run_descriptions` / `last_run_description_errors`, and `/health/auto-summary` gained a `pending_descriptions` backlog count alongside the existing summary backlog. New tests in `tests/test_auto_summary_async.py` (description backfill: inject, skip-existing, defer-as-error, disabled-skip; plus the `pending_descriptions` surface) and a rewritten `TestSaveDescriptionPending` in `tests/test_description_graceful_degrade.py` pinning the new "never call the LLM inline" contract.

- **Consolidation + lint LLM calls now route through the centralized `OllamaClient` (#338 Phase 4).** `consolidation/runner.py::_call_llm_with_fallback` and `lint/contradictions.py::_call_llm_for_contradiction` were the last direct-`httpx` Ollama callers. They target an OpenAI-compatible `/v1/chat/completions` server, so the client gained a `chat_completions(messages, model, base_url=…, …)` method (CONSOLIDATION role) that returns the assistant message content and raises `OllamaError` on transport failure or a malformed response. The consolidation fallback chain stays in the runner — each attempt passes its own `base_url`/`model` (via the new per-call `base_url` override on the client) and uses `retries=0`, because the chain itself is the retry strategy and shouldn't re-hammer each slow (600 s) host. Behaviour preserved: the runner still walks primary → fallbacks and raises `RuntimeError` only when all fail; the contradiction check still returns `""` (→ `UNKNOWN`) on failure. The only remaining direct-`httpx` Ollama touch is the `/health` liveness probe in `api/server.py`, intentionally left raw (a liveness check shouldn't be gated by the circuit breaker) pending the `/status`+doctor metrics phase.

- **Embedding path now routes through the centralized `OllamaClient` (#338 Phase 3).** `embedder._embed_local` and `check_model_context` no longer build their own `httpx.post` — they call `OllamaClient.embed()` / `OllamaClient.show()` (EMBED role → the configured embedding host). `OllamaClient.embed()` now owns the full embed contract: the `/api/embed` → `/api/embeddings` legacy fallback (now triggered **only** on a 404 API-version mismatch, not on transient timeouts — slightly less wasted latency on a down host), both response shapes, and context-overflow detection. The public contract is unchanged: `embed()` returns `[]` on connectivity/timeout failure (the signal the indexer uses to skip + retry) and raises `EmbeddingContextError` on a context-window overflow. `EmbeddingContextError` and `_is_ctx_overflow_message` moved to `palinode.core.ollama_client` and are re-exported from `palinode.core.embedder` for backward compatibility. Per-call embed logging now lands on the structured `palinode.ollama.events` JSON surface (with embed I/O gaining retry, the circuit breaker, and rolling latency/error metrics for free); the embedder keeps a single privacy-preserving summary WARNING (`text_len`, never raw text) when an embed degrades to `[]`. `OllamaError` gained a `status_code` attribute so the embed path can tell a 404 from other HTTP errors.

- **Auto-description and auto-summary now route through the centralized `OllamaClient` (#338 Phase 2).** `_generate_description` and `_generate_summary` in `palinode/api/server.py` no longer build their own `httpx.post` — they call `get_ollama_client().generate(..., role=CHAT, retries=0)`. Behaviour-preserving for the happy path, with two operational gains: (1) routing the inline description through the per-role **circuit breaker** means a chat-host brownout fast-fails to `description_pending`/deferred instead of spending the full 5 s timeout on *every* save; (2) circuit-open is now treated like a timeout (defer + watcher-retry), and a malformed Ollama body surfaces as a typed `OllamaError` (fallback) rather than a bare `JSONDecodeError`. `retries=0` keeps both paths single-shot so the #336 latency contract holds. The centralized client also now wraps a non-JSON response body in `OllamaError` so no caller ever sees a raw decode error.

### Fixed

- **auto_summary / auto-description output quality on small instruct models (#338 / auto_summary UX).** Some small instruct models routinely ignored the prompt's length cap — emitting 200+ char "one-sentence" results that the old code chopped mid-word — and prepended meta-preamble ("The memory describes …", "Here's the summary:") despite the "no preamble" instruction. Two fixes: the description/summary prompts now explicitly forbid those lead-ins and restate the hard character limit; and a new `_clean_llm_oneliner` post-processor strips the common preamble/label lead-ins and clips overshoot at a sentence/word boundary (with an ellipsis) instead of truncating mid-token. Conservative — legitimate sentence subjects (e.g. "The system decided …") are left untouched. Eight `_clean_llm_oneliner` unit tests plus updated description/summary tests in `tests/test_description_graceful_degrade.py` / `tests/test_description.py`.

- **MCP write-path timeouts now tell the model to verify before retrying (#416).** When `palinode_save` or `palinode_session_end` exceeded the request timeout, the MCP layer returned a generic `Error: Request to ... timed out.` — but the durable write often succeeded server-side (a slow `auto_summary` or embedding step outlasting the timeout). That misled callers into blindly retrying and creating duplicate entries. These two write-path tools now return a `Timeout:`-prefixed message that says the write may have committed and instructs the caller to `palinode_search` for a distinctive phrase before retrying. Read-path tools keep the plain timeout message. The `palinode_save` tool description mirrors the same hint so the model rationalizes the verify path. The `Timeout:` prefix is recognized by the MCP call-audit error detector so timed-out writes are still logged as errors. Tests in `tests/test_mcp.py` cover the message contract and drive the async dispatcher with a mocked slow server.

## [0.8.12] — 2026-05-25

Targeted hot-path patch tail to 0.8.11. Headline fix: `auto_summary` is no longer called synchronously inside `/save`. The inline LLM call could block the response for the model's full first-token latency; on a cold or contended Ollama this presented to REST consumers as a "write timeout" while writes were succeeding server-side. The release also delivers `/health/auto-summary` and a `/status.auto_summary` block so monitor agents can detect a stalled async summary pipeline, bounded snippets on `/search-associative`, and a `recallProfile` config field with five named presets in the openclaw-palinode plugin.

### Changed

- **`auto_summary` is now async — moved off the `/save` hot path.** Previously `/save` called `_generate_summary` inline whenever a file qualified (`core:true`, no `summary`, content >= `min_content_chars`), blocking the response for the LLM's full first-token latency. On a cold or contended Ollama this could turn into a multi-second wait; REST consumers with conservative timeouts could see "write timeouts" while the write was actually succeeding server-side. The watcher already debounced `/generate-summaries` calls for files matching the same criteria (`palinode/indexer/watcher.py::_schedule_summary_generation`), so the fix is small: drop the inline call, surface `summary_pending: true` in the `/save` response when the file qualifies (mirroring the existing `description_pending` pattern), and let the watcher path do the LLM work out of band. Embeddings are unaffected — they still run inline.

### Added

- **`/health/auto-summary` endpoint.** New auth-exempt health check parallel to `/health/watcher`, designed for external monitor agents to detect a stalled async summary pipeline. Returns `status: "ok" | "degraded" | "down"` with a `reason` field. Probes the auto_summary Ollama URL (which may differ from the embed URL), scans for pending files (`core:true`, no summary, body >= threshold) capped at 1000, and applies a decision tree: `down` if Ollama unreachable; `degraded` if pending backlog ≥ 50 OR last run had errors with non-zero pending OR last run > 30min old with non-zero pending; `ok` otherwise. Disabled `auto_summary` always returns `ok` with a clarifying `reason`.
- **`auto_summary` block in `/status`.** Surfaces `last_run_at`, `last_run_duration_ms`, `last_run_count`, `last_run_errors`, `last_error`, `total_runs`, `total_errors` from a new module-level `_auto_summary_state` populated by `/generate-summaries` on every run. Mirrors the shape of the existing `reindex` and `write_time_*` observability blocks. Eleven regression tests in `tests/test_auto_summary_async.py`.

### Fixed

- **`/search-associative` now per-result snippet-enriches its response.** The associative spreading-activation endpoint was overlooked when `_enrich_with_snippets` landed for `/search` and `/search-recent`. A single associative hit on a multi-fact aggregated file could still return tens of KB of un-truncated `content`, defeating the budget guarantee that the rest of the search surface honours. Fix: one call to `_enrich_with_snippets(results, req.query, config.search.snippet_max_chars)` in `search_associative_api` mirrors the `/search` pattern. `content` is preserved untouched for API/CLI consumers; `snippet` and `content_truncated` are added per result. Three-test regression suite in `tests/test_search_associative_snippet.py` pins the behaviour.

### Plugin

- **openclaw-palinode: `recallProfile` config field + five named presets.** autoRecall was previously a binary switch with hardcoded four-source injection. A short prompt could expand into tens of thousands of tokens of effective input when semantic search matched large incident-style memory files and the model proceeded to reason about that prior incident rather than execute the current request. Size *and* modality were both wrong. Fix: named recall profiles (`coding`, `monitoring`, `investigation`, `writing`, `conversation`, plus `minimal` / `off`) compose source enablement, per-source caps, type allow/deny, and a total-budget hard cap. `coding` preserves existing behaviour (default if unspecified). `monitoring` ships triggers only, denies `RCA`/`Postmortem`/`Incident`/`Reflection`, caps at 3K. `recallProfileConfig` shallow-overrides individual fields without forking a preset. `autoRecall: false` still works (forces `off`). Plugin also switches semantic + associative result rendering to prefer the server's query-windowed `snippet` field over blunt `r.content.slice()`, with a fallback for older servers. Wrapping element now includes the profile name (`<palinode-memory profile="...">`) and the info log line surfaces profile + sources + final injection size for observability. Type filters (`type_allow` / `type_deny`) are forwarded defensively to `/search` and `/search-associative`; older Palinode servers ignore unknown body keys. Seven unit tests in `plugin/test/recall-profile.test.ts` pin the catalog shape and override semantics.

M1 hardening tail + audit-driven critical-bug remediation. 13 issues closed across leak-regression guards (#341, #275-prep), MCP registry hygiene (#313), doctor FTS5 sync surfacing (#316), cross-surface session-end timeout (#377), /wrap push semantics (#353), embedder logging (#383), four silent-failure-returns-success criticals (#384–#387), Ollama context-window hardening (#335), and graceful-degrade auto-description (#336). Two audit deliverables shipped on dedicated branches (#337 logging, #378 session-end). One investigation comment drafted (#373 Claude Desktop). #338 (centralized Ollama client) explicitly deferred to v0.8.12.

### Changed

- **`upsert_chunks` return contract: `int` → `dict{written, vec_ok, fts_ok}` (#385).** The function previously returned a bare integer (chunks written). It now returns a dict so callers can detect per-index health without a separate query. `index_file` and `POST /save` both surface `indexed_vec` and `indexed_fts` from this result. MCP `palinode_save` warns when either flag is False.

### Fixed

- **OpenClaw autoRecall memory now uses system-role framing (#358).** The plugin extracts the `<palinode-memory profile="...">` composer into a single exported helper and returns recalled memory through `systemContext` instead of the flat user-prepend channel; the fallback composer uses explicit `reference_context` / `user_instruction` delimiters for hosts without a system surface.

- **`upsert_chunks` vec0 and FTS5 write failures are now logged and surfaced (#385 / C3).** Both the pre-INSERT DELETE and the INSERT-after-DELETE pair on `chunks_vec` were wrapped in bare `except Exception: pass`. A vec0 structural failure silently produced a `chunks` row with no corresponding `chunks_vec` row — the chunk was FTS5-searchable but invisible to vector search. Fix: vec0 final-retry failures log at ERROR with `exc_info=True`; FTS5 sync failures log at WARNING; the INSERT-first FTS5 pattern avoids the "malformed" DELETE-on-empty bug.
- **`POST /save` now returns `git_committed`, `indexed_vec`, and `indexed_fts` (#386 / C4, #385).** The git auto-commit block logged at error level but the response had no field — `git_committed: false` was invisible to callers. - **Consolidation runner YAML parse failures now log and count instead of silently skipping (#387 / C5).** `_collect_daily_notes` and `_get_decisions_for_project` both had `except Exception: pass/continue` on YAML parse errors. Corrupt frontmatter silently dropped files from consolidation. Fix: both log WARNING with filepath, exception, and recovery hint ("run `palinode lint`"). `_collect_daily_notes` returns `(notes, skipped_count)`; callers include `yaml_parse_errors` in the run summary when non-zero. Body text is still collected even when frontmatter fails.
- **`recent_save_embeddings` DB open failure now logs WARNING instead of silently returning `[]` (#384 / C2).** A missing or corrupt DB caused the dedup window to silently disable, making every save appear unique. Now logs at WARNING with `db_path` and exception so operators can correlate dedup misses with infrastructure issues.

- **`palinode/core/embedder.py` now has a logger; all Ollama failure paths surface (#383 / C1).** The module had no logger at all — every embed failure (timeout, HTTP error, connect error, unexpected response shape) was silent, forcing operators to correlate downstream symptoms (zero search hits, dedup misses, Phase 7 smoke flakes) with no log context. Added `logger = logging.getLogger(__name__)` and per-failure log calls: each exception class (TimeoutException, HTTPStatusError, ConnectError) logs at WARNING with `exc_info=True` and structured context (model, endpoint, text_len, timeout setting, retry index). Unexpected-shape 200 responses log at WARNING with `response_keys`. Successful embed logs at DEBUG with timing. Terminal all-endpoints-exhausted case logs at WARNING with model, url, text_len, and timeout_seconds — the exact context needed to correlate with smoke-rig reports. Raw text is never logged; only `text_len` appears so PII does not reach log aggregators.
- **Session-end timeout raised to 90s on all three surfaces (#377).** The 30s CLI timeout and 10s hook curl timeout were too short for multi-decision payloads (BGE-M3 dedup embed + git commit on a remote private-network host). All three surfaces now use a shared `SESSION_END_TIMEOUT_SECONDS = 90.0` constant from `palinode.core.defaults` (overridable via `PALINODE_HOOK_TIMEOUT` / `PALINODE_SESSION_END_TIMEOUT` env vars). CLI, MCP, and hook all import or reference the same source of truth; module-load assertion guards in `cli/_api.py` and `mcp.py` catch future drift at import time. Hook script `PALINODE_HOOK_TIMEOUT` default is 30s; `settings.json` runner timeout raised to 35s to stay ahead of curl. CLI now distinguishes `ReadTimeout` (slow API, recoverable) from `ConnectError` (API down) with separate error messages. Eight new regression tests in `tests/test_session_end_timeout.py`.
- **`/wrap` now pushes to remote before archiving (#353).** The `/wrap` slash command calls `palinode_push` (MCP tool) before `palinode_session_end`, so local commits are on the remote before the daily note is written and context is cleared. Skips silently when there are no unpushed commits or no remote is configured; surfaces a clear message on push failure so Paul can decide whether to proceed. `palinode init`'s `WRAP_COMMAND_BODY` is updated to match.
- **Embed path hardened against Ollama context-window mismatch (#335).** When Ollama returns a 200 OK with `{"error": "prompt is too long for max context"}` (its silent-truncation failure mode when `num_ctx` is too small), `_embed_local` now raises `EmbeddingContextError` — a typed exception with `model`, `text_len`, and `ollama_message` attributes and a recovery hint pointing at the modelfile fix. Added a lazy preflight check (`check_model_context`) that queries `/api/show` on the first embed call and logs a WARNING if `num_ctx` is below 8192 (bge-m3's actual capability vs. Ollama's 4096 default — the exact regression from 2026-05-04). Preflight runs once per process (thread-safe) and never blocks embed on failure.
- **`palinode_save` auto-description now has a hard timeout with graceful degrade (#336).** Previously the description-generation call to Ollama used a hardcoded 15 s timeout with no degrade path — a cold-loading or overloaded Ollama would stall every `/save` for the full duration. Now: (1) `_generate_description` has a configurable timeout (`config.auto_summary.describe_timeout_seconds`, default 5 s, override via `PALINODE_DESCRIBE_TIMEOUT_SECONDS`) that returns a `_DESCRIPTION_DEFERRED` sentinel on `TimeoutException`; (2) the `/save` handler detects the sentinel, saves the file without a description, and includes `description_pending: true` in the response so callers know to expect a fill-in; (3) the watcher detects files with no `description` field after any file event and schedules a debounced retry via `/generate-summaries` (10 s window, accumulates rapid saves). The audit Q2 finding is also fixed: non-timeout description failures (e.g. `ConnectError`) were logged at `INFO`; they now log at `WARNING`.

### Added

## [0.8.8] — 2026-05-01

A pure-security release in response to the MCP Marketplace security audit.
Closes 27 of the 30 findings (5 critical, 8 high, 12 medium, 2 low). No
breaking changes; all hardening is additive. Local-first deployments keep
working without ceremony.

### Security

- **Optional bearer-token authentication on the API server.** Set
  `PALINODE_API_TOKEN` (or `PALINODE_API_TOKEN_FILE` for docker-secrets
  style deployments) to require `Authorization: Bearer <token>` on every
  request other than `/health` and `/health/watcher`. Off by default to
  preserve zero-friction local dev. The startup gate now refuses to launch
  when `PALINODE_API_BIND_INTENT=public` is set without a token, closing
  the public-bind hardening gap. Comparison uses `hmac.compare_digest` to
  remove the timing side-channel. The gate fires at module import so it
  triggers under `uvicorn palinode.api.server:app` (the canonical systemd
  ExecStart pattern), not just from `main()`.
- **Bearer-token authentication on the MCP HTTP transport (#289).** The
  MCP HTTP server (port 6341) now enforces the same `Authorization: Bearer
  <token>` requirement as the API server when `PALINODE_API_TOKEN` is set —
  closing the gap where an authenticated API was reachable unauthenticated
  via the MCP HTTP surface. Token source is shared (`PALINODE_API_TOKEN` /
  `PALINODE_API_TOKEN_FILE`): one deployment, one secret. A separate
  `PALINODE_MCP_BIND_INTENT=public` gate (distinct from
  `PALINODE_API_BIND_INTENT`) refuses to start the MCP HTTP server when
  publicly bound without a token. The gate lives **inside `main_http()`**
  (not at module import) so importing `palinode.mcp` for the stdio
  transport — which never calls `main_http()` — does not trigger it.
  `/healthz` is exempt so uptime probes don't need the token. The shared
  auth primitives (`BearerAuthMiddleware`, `load_api_token`,
  `validate_auth_config`) are extracted into `palinode.core.auth` so
  neither server duplicates the security-sensitive comparison logic.
- **Dependency lower-bounds added** to exclude 15 known CVEs against
  `pyyaml`, `httpx`, `sqlite-vec`, `fastapi`, `uvicorn`, `pydantic`, and
  `mcp`. All currently-installed versions are already past the fix
  windows; this codifies that floor.
- **Path traversal hardening** in `_resolve_memory_path` migrated from
  `os.path.realpath`/`commonpath` to `pathlib.Path.resolve(strict=True)`
  + `Path.is_relative_to()`. Generic `"Invalid path"` returned to clients
  (no filesystem-info leak); original input logged at INFO for operators.
- **Streaming body-size enforcement.** New ASGI middleware counts bytes
  per chunk and rejects with HTTP 413 mid-stream. Closes the
  chunked-encoding bypass that the previous header-only check missed.
- **Wiki footer entity validation.** Slugs must match `^[A-Za-z0-9._-]+$`
  before being emitted inside `[[wikilinks]]`; malformed slugs are
  dropped with a warning rather than corrupting markdown.
- **LLM prompt content** in `_generate_description` and `_generate_summary`
  now wrapped in `<user_content>` delimited tags with neutralization of
  injected close tags. Best-effort defense against prompt-injection.
- **Subprocess argv-form audit** confirmed all `subprocess.run` calls in
  the codebase use list-of-args, never `shell=True`. AST-based test in
  `tests/test_subprocess_argv_form.py` fails CI on any future regression.
- **CORS wildcard rejection** at startup. `PALINODE_CORS_ORIGINS=*` now
  refuses to launch with a clear error; each origin must parse as a
  valid http/https URL with non-empty netloc.
- **Secret-redacting logging filter** prevents API keys (`sk-...`,
  `xoxb-...`, AWS keys) and basic-auth-in-URL credentials from leaking
  via `logger.exception()` tracebacks if a memory file contains them.
- **Rate limiter dict bounded.** `_rate_counters` now prunes expired
  entries on every check and caps at `PALINODE_RATE_LIMIT_MAX_KEYS`
  (default 10000) with oldest-window eviction. Prevents memory growth
  under varied-IP scans.
- **TOCTOU mitigation** in file reads. New `_open_memory_file_text()`
  helper uses `os.open(..., O_RDONLY | O_NOFOLLOW)` so symlink swaps
  raise `OSError` rather than silently following the link.
- **Narrowed broad `except Exception` blocks** at LLM, HTTP, and git
  call sites. Specific exception types (`httpx.HTTPError`, `OSError`,
  `json.JSONDecodeError`, etc.) make root-cause tracing easier and
  prevent operational errors from masking security signal.

### Fixed

- CI security-scan job: bandit findings annotated with `# nosec` plus
  rationale rather than suppressed wholesale. `pip-audit` now runs (was
  blocked by the bandit failure in v0.8.5).
- `INSTALL-CLAUDE-CODE.md` tool count corrected (21 → 25) and four
  previously-undocumented tools added to the table:
  `palinode_cluster_neighbors`, `palinode_topic_coverage`,
  `palinode_depends`, and `palinode_timeline` (deprecated alias).

### Internal

### Fixed

- **MCP tolerates JSON-encoded array arguments (#147)** — some MCP transports double-encode array args as JSON strings. Previous behavior rejected with Pydantic 422 ("expected array, received string") on `palinode_session_end` and `palinode_save`, and silently corrupted the `paths` filter on `palinode_diff` (`",".join(string)` was joining char-by-char). New `_coerce_str_array` helper decodes JSON-array strings while passing native lists through unchanged.
- **SessionEnd hook reason filter (#149)** — the hook's README claimed a `clear` matcher that never existed in `settings.json`. Replaced with explicit script-side filtering via a new `PALINODE_HOOK_REASONS` env var. Default allowlist: `clear logout prompt_input_exit other`. Override to narrow (e.g. `"clear"`) or extend (`+resume`, `+bypass_permissions_disabled`).
- **Empty-transcript captures (#151)** — SessionEnd hook was capturing 0-message sessions despite `PALINODE_HOOK_MIN_MESSAGES=3`. Root cause: `grep -c '.'` on empty input emitted `"0"` and exited 1; the `|| echo "0"` fallback then appended another `"0"`, producing the literal three-byte string `"0\n0"` which made the integer test error and fall through. Fix: drop the `|| echo "0"` (grep always emits an integer regardless of exit), use `|| true` to absorb the non-zero, default to `0` if the pipeline produces nothing.
- **Test isolation** — full-suite test runs no longer hit 5 failures in `test_write_time` due to scope-chain reload state pollution.

### Documentation

- **SSH keepalive options for remote-stdio MCP (`docs/MCP-SETUP.md`, `docs/INSTALL-CLAUDE-CODE.md`)** — `ServerAliveInterval=30`, `ServerAliveCountMax=3`, and `TCPKeepAlive=yes` added to the example configs. This helps long-lived SSH-backed MCP sessions survive idle network drops and reconnect more cleanly after laptop sleep or WiFi changes.

### Tests

- 325 tests passing at release time.

## [0.5.0] — 2026-04-10

First tagged release. Persistent memory for AI agents with git-versioned markdown as source of truth, hybrid SQLite-vec + FTS5 search, and LLM-driven consolidation applied by a deterministic executor.

### Added

**Core storage and search**
- SQLite-vec vector store with BGE-M3 embeddings (1024d) via any OpenAI-compatible endpoint (Ollama, vLLM, etc.)
- Hybrid search: vector similarity + BM25 (FTS5) fused via reciprocal rank fusion
- Hash-on-read freshness validation — detects out-of-band file edits without a full reindex
- File watcher daemon with debounced reindex and fault isolation

**Consolidation and compaction**
- Deterministic executor applying `KEEP` / `UPDATE` / `MERGE` / `SUPERSEDE` / `ARCHIVE` operations proposed by an LLM (LLM proposes structured operations, deterministic Python applies them — keeps every edit reviewable in git)
- Weekly full-corpus consolidation with configurable LLM backend
- Nightly lightweight consolidation pass (`--nightly` flag) bounded to `UPDATE`/`SUPERSEDE` for safer incremental updates
- Model fallback chains — primary → fallback → fallback on timeout or HTTP error
- Prompt versioning system — extraction/compaction prompts stored as memory files with `active: true` frontmatter

**Interfaces (all four expose the same capabilities)**
- **MCP server** — Streamable HTTP transport (also supports stdio) with 18 tools. Stateless HTTP client, point it at any Palinode API server
- **REST API** — FastAPI on port 6340, 20+ endpoints covering search, save, diff, triggers, history, blame, rollback, consolidation, session-end, lint, migrate
- **CLI** — 26 commands wrapping the REST API via Click. TTY-aware (human output interactive, JSON when piped). Remote access via `PALINODE_API` env var
- **Plugin** — OpenClaw lifecycle hooks for agent frameworks with inject/extract patterns

**New MCP tools in this release**
- `palinode_lint` — scan memory for orphaned files, stale active files, missing frontmatter, potential contradictions
- `palinode_blame` / `palinode_history` / `palinode_rollback` — git-backed provenance tools
- `palinode_trigger` — register prospective triggers that inject memory files when matching context is detected
- `palinode_prompt` — list, read, and activate versioned LLM prompt files

**Capture and session management**
- `/session-end` endpoint — appends session summary to daily notes + one-liner to project status files

- Entity extraction from daily notes with keyword fallback for untagged content

**Migration**
- `palinode migrate` — import existing markdown memory systems (OpenClaw format) with `--review` mode for dry-run inspection

**Security and hardening**
- Path validation on all file operations (rejects `..`, symlinks outside memory directory)
- Secret scrubbing on save path via configurable regex patterns

**Documentation**

## [0.1.0] — 2026-03-22

Initial public release. Minimum viable memory system.

### What worked at launch

- SQLite-vec vector store with BGE-M3 embeddings via Ollama
- File watcher daemon auto-indexing markdown on create/modify/delete
- FastAPI server with `/search`, `/save`, `/status`, `/reindex` endpoints
- Markdown parser with YAML frontmatter extraction and heading-level section chunking
- CLI with `search` and `stats` commands
- Plugin with core memory injection, topic-specific retrieval, and three tools (`palinode_search`, `palinode_save`, `palinode_status`)
- Session capture to daily notes on agent end
- Systemd user services for `palinode-api` (port 6340) and `palinode-watcher`

### Architecture decisions

| Decision | Choice | Rationale |
|---|---|---|
| Source of truth | Markdown files, git-versioned | Human-readable, survives everything |
| Vector store | SQLite-vec (embedded) | No server, matches file-based philosophy |
| Embeddings | BGE-M3 via Ollama | Local, private, strong on structured text |
| Prompts | Read from `specs/prompts/*.md` | Version-controlled, editable, diffable |

### Known limitations at 0.1.0

- No WAL mode on SQLite — concurrent writes can lock
- Session capture wrote raw text to daily notes (no LLM extraction)
- No consolidation scheduler
- No entity linking
- No MCP server (plugin-only integration)

All of these are addressed in 0.5.0.
