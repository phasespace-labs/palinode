## [0.8.14] ŌĆö 2026-06-24

The memory-architecture release. Two themes land together. **The provenance wedge** opens with a read-only, loopback-only local UI (`/ui`, Phases 0+1 ŌĆö dashboard, browse/search, diffs, compaction review, quality queues, per-fact provenance panel) and a source-citation capture path (`sources:` quote+hash anchors, #459) ŌĆö the audit-grade-memory read surface, all plaintext, shipping ahead of any signed layer. **The ADR-015 write-semantics axis** lands the `update_policy` append/replace declaration (#431), TTL / auto-archive for ephemeral monitoring memories (#482), telemetry recall-exclusion (#398), incident-status lifecycle validation, and the deterministic-executor guards that stop consolidation from forking a living document (SUPERSEDE/ARCHIVE/RETRACT refused on `update_policy: replace`). The recall-feedback loop is made durable end-to-end ŌĆö the #371 write-back is grounded into ADR-007 demand-decay importance (#86) and hardened against four adversarial defects (recall-stat pollution, re-index metadata wipe, RETRACT fork, metadata-bypass validation). Under the hood, `api/server.py` is fully decomposed into router modules (server is now pure assembly, #314) alongside ranker / op-parse / enrichment / `_shared` extractions, and **every memory-file mutation now routes through one atomic choke point with per-file commits** (#564/#565/#566/#567) ŌĆö pure substrate consolidation through one auditable write path. Plus the formal executor-determinism spec (#312), an OpenAI-compat CHAT fallback chain (#464), `session_end(push=true)` one-call note-ship (#378), `init --skills` (#474), and a `fastapi<0.137` pin closing the routeless-API regression (#544).

### Added

- **MIT `LICENSE` file at repo root.** `pyproject.toml` and the README already declared MIT, but no `LICENSE` file backed the claim ŌĆö so GitHub's repo-metadata sidebar showed no license. Adds the canonical MIT text (┬® 2026 Paul Kyle). Ships publicly (not a scrub-forbidden path).

- **Local read-only provenance UI ŌĆö Phase 0 (`/ui`).** A server-rendered, loopback-only web view of memory health and per-fact provenance, mounted on the existing API (`palinode-api`) under `/ui` ŌĆö no new service, no build step, no CDN (templates + CSS ship in-package, offline-first). Phase 0 ships two views: a **Dashboard** (`GET /ui`) summarising memory health (total memories, chunks, core count, stale/orphaned/no-description/contradiction counts, plus 7-day commit count and a recent-memory list) and a **Fact detail** view (`GET /ui/memory/{path}`) that renders a memory's markdown body and a vertical **provenance panel**. The dashboard's "memories" count is the on-disk markdown-file count from the same `lint` scan that produces the orphaned/stale/no-description counts (markdown is the source of truth), so the figures stay coherent regardless of index state; "chunks" stays the indexed-DB metric and an "unindexed ŌĆö start the watcher" hint explains a fresh/empty index. The UI is a pure client of existing capabilities ŌĆö it introduces no business logic the store/API doesn't already expose (status/lint/read/git lineage/recall stats). Memory bodies are rendered server-side with `markdown-it-py` (raw HTML disabled) and then sanitized with `nh3` (an XSS backstop ŌĆö this is an audit tool rendering agent-generated content). The provenance panel shows what is verifiable today honestly ŌĆö git lineage (the Saved commit), supersession (frontmatter `supersedes`), and retrieval stats (recall count) ŌĆö with muted "not yet captured" placeholders for the attestation / identity / trusted-timestamp / source-span / contradiction / recall-session fields that the data does not carry yet. **Loopback-only:** `/ui` hard-refuses (HTTP 403) when the API is bound to a non-loopback address; it is unauthenticated and read-everything, so there is no safe public path in P0. **Read-only:** no mutations. The markdown-render + sanitize + provenance-panel helpers are factored as store-agnostic modules (`palinode/api/ui/render.py`, `palinode/api/ui/provenance.py`) so other consumers can reuse them. New deps: `jinja2`, `markdown-it-py`, `nh3`. New `tests/test_ui_provenance.py` (real SQLite + tmp_path) covers dashboard render, fact render, markdown `<script>` sanitization, the loopback refusal, and the helper units.

- **Local read-only provenance UI ŌĆö Phase 1 (browse / search / diffs / compaction / quality).** Wires the four previously-inert sidebar nav items into working views on the existing `/ui` router (still read-only, loopback-only, no new service, no build step, reusing the Phase 0 design system). **Memory list** (`GET /ui/memory`) ŌĆö every browsable memory on disk (file-sourced, so it stays coherent with the dashboard's file-based counts on an unindexed store) with type / core / freshness filters, each row linking to its fact-detail page. **Search** ŌĆö a search box on the memory list (`GET /ui/memory?q=`) that runs the existing `search_api` in process (no new ranking logic) and degrades to a soft "search unavailable" banner instead of a 500 when the embedder is unreachable. **Diffs** (`GET /ui/diffs`) ŌĆö recent memory changes from git, grouped by day. **Compaction review** (`GET /ui/compaction`) ŌĆö the last consolidation passes, read from git history (commits whose subject marks them a compaction/nightly pass) plus the `-history.md` audit-trail siblings; it reads history rather than triggering consolidation. **Quality queues** (`GET /ui/quality`) ŌĆö `lint` findings (stale, orphaned, no-description, contradictions) plus a new missing-extraction-metadata bucket, each item linking to its fact. A new read-only `git_tools.recent_commits(days, limit, message_prefix)` provides the repo-wide commit list (counterpart to the per-file `history`); the view-shaping helpers live in a store-agnostic `palinode/api/ui/views.py` so other consumers can reuse the list/search/diff/quality shaping. A single `is_browsable_memory` predicate (excludes `-history.md` consolidation siblings and the skip-dirs) is the source of truth every memory surface agrees on ŌĆö the sidebar memory badge, the dashboard MEMORIES card, the memory list, and the Quality queues all filter through it, so a `-history.md` sibling can't inflate the count or get falsely flagged as needing attention. The Diffs and Compaction views scrub each commit's touched-file list to memory files only (an `is_memory_file` predicate dropping `.palinode.db`, its `-journal`/`-wal`/`-shm` sidecars, and anything under `logs/` ŌĆö the `git add -A` known-issue stages these); a commit with zero memory files after scrubbing is dropped from the Diffs view. New `tests/test_ui_phase1.py` (real SQLite + tmp_path) covers each view rendering seeded data, search hits + graceful degradation, the `-history.md`-not-counted/not-flagged coherence, DB/log scrubbing in diffs+compaction, and the helper units.

- **Source-citation capture path ŌĆö `sources:` quote anchors (#459).** Completes the capture half of the source-citation feature (the verifier, `palinode.core.quote_verify`, landed earlier). Saves can now carry an optional `sources:` list of `{ref, quote, quote_hash}` anchors ŌĆö each pins a memory to the exact passage it cites in another file. Exposed across all four surfaces with ADR-010 parity: `sources` on the `/save` API (`SaveRequest`), the `palinode_save` MCP tool (array-of-objects schema), the OpenClaw plugin (TypeBox), and `palinode save --cite 'REF::QUOTE'` on the CLI (repeatable; split on the first `::`). On save the `quote_hash` is auto-computed from the normalized quote when omitted, or validated against the quote when supplied ŌĆö a stored hash that doesn't match its quote is an inconsistent anchor and is rejected (HTTP 400), as is any malformed entry (non-list, missing/empty `ref` or `quote`). Frontmatter stays clean when no anchors are given. The existing verifier is now wired into **`lint`** (`run_lint_pass` ŌåÆ `source_anchor_issues`, surfaced on the API `/lint` route and the CLI `palinode lint` report): each file's anchors are checked for drifted / source-missing / tampered status, a clean no-op for files with no `sources:`. `palinode/core/parser.py` gains a typed `parse_sources` accessor. New `tests/test_source_capture_g1.py` (real SQLite + tmp_path) plus three new plugin parity assertions (`npm test`).

- **Config-driven MCP tool advertisement surface (#499).** `tool_surface` / `PALINODE_MCP_SURFACE={core,full}` now controls which MCP tools `list_tools()` advertises. The default remains `full` for upgrade compatibility; `core` advertises the hot-path subset while dispatch capability remains unchanged.

- **Formal consolidation executor determinism spec (#312).** Added `docs/EXECUTOR-SPEC.md` as the public behavioral contract for `apply_operations`: operation preconditions/postconditions for KEEP, UPDATE, MERGE, SUPERSEDE, ARCHIVE, and RETRACT; ordering and idempotency rules; malformed-op, replace-policy, and nightly-merge rejection behavior; history preservation; atomic write semantics. This is the test oracle for #311 and the reference contract for future ports/integrations.

- **`session_end` can ship its own note in one call ŌĆö `push` parameter (#378).** `palinode_session_end` / `POST /session-end` / `palinode session-end --push/--no-push` / the `palinode_session_end` MCP tool now accept an optional `push` (bool): `true` pushes the memory repo immediately after committing the session note, `false` suppresses the push even when `auto_push` is on, and omitted defers to `config.git.auto_push` (the legacy default, off ŌĆö so existing callers are unchanged). The `/session-end` response gains `committed` and `pushed` booleans so a caller can report "saved + pushed" versus "saved ŌĆö note committed locally but NOT pushed" honestly. Because `git push` is repo-wide, a single `session_end(push=true)` also ships any earlier same-session `/save` commits. This turns the `/wrap` note-ship from a forgettable trailing prose step into a structural property of the tool (see Changed). ADR-010 parity across API / MCP / CLI; plugin parity not required (`session_end` is not a plugin surface). New tests in `tests/test_session_end.py` cover the push decision matrix (param-true / default-off / param-none-honors-auto_push / param-false-overrides) and honest failure reporting.

- **TTL / auto-archive for ephemeral monitoring memories (ADR-015 ┬¦2.3, #482).** Completes the deterministic-monitor memory axis. A memory can now carry an expiry via `metadata.ttl` (a duration ŌĆö seconds or `<N>{s,m,h,d,w}` ŌĆö resolved to an absolute `expires_at` at save time) or an explicit `expires_at` ISO-8601 timestamp; a malformed value is rejected (HTTP 400). A new deterministic, idempotent **`archive-expired` sweep** flips every memory whose `expires_at` has passed to `status: archived`, so it ages out of default recall while staying on disk + in git for audit ŌĆö leveraging #485's archived-status recall-suppression. Because a frontmatter-only status change doesn't move the body content-hash that `index_file` keys its fast path on, the sweep also calls the new `store.set_status_for_path()` to push the status into the chunk index directly (no body re-embed). Exposed with ADR-010 parity across the maintenance-op surfaces ŌĆö CLI `palinode archive-expired [--dry-run]`, `POST /archive-expired`, and the `palinode_archive_expired` MCP tool (plugin-exempt, mirroring `consolidate`); intended for cron / the monitor harness. `daily/` notes are exempt; the sweep is idempotent (already-archived files are skipped). New `tests/test_ttl_archive_482.py` (real SQLite + tmp_path). The secondary `source:<probe>-deterministic` provenance hint (ADR-015 ┬¦2.3) is deferred to a follow-up.

- **Human-assigned memory priority (#259).** Saves can now carry optional `priority: 1..5` frontmatter, exposed across the API, CLI (`--importance N`, `--important`, `--critical`), and MCP surfaces (plugin parity tracked in #480). Missing priority is normal (`3`), `min_priority` filters search results after fetch, and hybrid ranking gets only a small bounded additive nudge (`_PRIORITY_RANK_WEIGHT`) so relevance remains dominant. The existing ADR-007 `chunks.importance` demand-decay float remains system-assigned and unchanged.

- **`palinode mcp-smoke claude-desktop` warns when Claude Desktop is running (#373).** Completes the code half of #373 (the docs half ŌĆö the quit-before-editing warning in `docs/MCP-CONFIG-HOMES.md` and `palinode mcp-config --diagnose` ŌĆö shipped in #435). Claude Desktop holds `claude_desktop_config.json` in memory and rewrites it on quit, silently clobbering any edit made while it runs (and stripping `url`-form MCP entries). A new best-effort `claude_desktop_running()` probe (argv-form `pgrep`/`tasklist`, no new dependency) matches the platform *application* ŌĆö the macOS `Claude.app` bundle path, the Windows `Claude.exe` image ŌĆö so it can never false-positive on the Claude **Code** CLI or a `claude` shell alias. When `palinode mcp-smoke claude-desktop` is run and the app is detected live, it prints a loud "quit before editing ŌĆö edits get overwritten on quit; recovery order quit ŌåÆ edit ŌåÆ relaunch" warning to stderr (so a piped runbook stays clean). Detection is conservative: an undetermined result (unknown platform, probe missing/errored) stays silent rather than crying wolf. New `TestClaudeDesktopRunning` / `TestClaudeDesktopWarning` in `tests/test_cli_mcp_smoke.py` cover the returncodeŌåÆ`bool|None` mapping and that the warning fires only for `claude-desktop` when running is `True`.

- **`palinode init --skills {none|project|personal|both}` ŌĆö install /save /ps /wrap as Claude Code *skills* (#474).** Current Claude Code merged user-scope commands into skills and no longer searches `~/.claude/commands/`, so a project-scope `/wrap` was only available in the inited repo. `--skills personal` installs the session commands to `~/.claude/skills/` so `/wrap` is typeable in **every** project; `project` writes `.claude/skills/<name>/SKILL.md`; `both` does both. Default `none` keeps existing behaviour unchanged (the legacy `.claude/commands/` scaffold under `--slash` is untouched). The skill bodies render from the same `*_COMMAND_BODY` constants as the slash commands (single source ŌĆö they can't drift), so `/wrap` carries the push-twice ordering (#465). Non-destructive: existing skills are skipped without `--force`. New `tests/test_init_skills.py`.

- **OpenAI-compat primary + fallback chain for the CHAT role (auto-description / auto-summary) (#464).** The per-memory description (`_generate_description`) and summary (`_generate_summary`) paths gain a configurable model ladder. **(1) Primary protocol** ŌĆö a new `auto_summary.api` selector: `"ollama"` (default, Ollama-native `/api/generate` ŌĆö unchanged) or `"openai"`, which drives the primary through an OpenAI-compatible `/v1/chat/completions` endpoint at `auto_summary.ollama_url` (LM Studio / vLLM / a shim ŌĆö e.g. an MLX qwen served by LM Studio on a Mac Studio). **(2) Fallback chain** ŌĆö `auto_summary.llm_fallbacks: list[{model, url}]` (mirrors `consolidation.llm_fallbacks`; **empty default = zero behavior change**), each an OpenAI-compat endpoint, walked in order on primary failure. With `api: "ollama"` the chain fires on a brownout (`OllamaTimeout` / `OllamaCircuitOpen`); with `api: "openai"` it fires on **any** primary failure (a remote primary's connect/HTTP drop is exactly what backups cover) and also when the primary returns empty/garbage. **(3) Egress cap** ŌĆö `auto_summary.llm_fallback_max_per_run` (default 10; `0` = unlimited) bounds how many deferred files may escalate to the chain (esp. its Sonnet link) in a single backfill walk. Reuses the existing `OllamaClient.chat_completions(..., base_url=...)` plumbing (nothing new in the client); the `generate()`ŌåÆ`chat_completions()` shape gap is bridged by wrapping the prompt as a single user message. Because both enrichment functions are reached **only** from the watcher-driven `/generate-summaries` backfill post-#405 ŌĆö never the `/save` hot path ŌĆö the whole ladder is structurally batched to the watcher cadence and `/save` latency is untouched whether the primary is healthy or down (no inline per-save egress). Example deployment ladder: Mac Studio MLX `qwen3-coder` primary ŌåÆ local vLLM `qwen-coder` ŌåÆ Sonnet shim. New `tests/test_chat_fallback.py` covers the OpenAI-compat primary (used / down-cascades / empty-cascades), the fallback walk + first-success-wins, description & summary cascade on any primary failure, the no-fallback and budget-exhausted deferral paths, and the per-run budget reset.

- **`update_policy` write-semantics axis (ADR-015 ┬¦2.1, #431 / #430 / #398).** A new optional save parameter and sticky frontmatter field, orthogonal to `type`, with two values: `append` (default ŌĆö today's episodic behaviour) and `replace` (the memory is a living/current-state document). Exposed across all four surfaces with ADR-010 parity: `update_policy` on the `/save` API (`SaveRequest`), the `palinode_save` MCP tool (enum), and `palinode save --update-policy {append,replace}` on the CLI (threaded through `cli/_api.py`). It is persisted as **sticky frontmatter** so the file declares its own regime ŌĆö the explicit param wins, otherwise a save that omits it inherits the value the file already carries; a save that never declares a policy writes no `update_policy` field at all. An unknown value is rejected (HTTP 400). **`append`'s default clobber behaviour is unchanged** ŌĆö a same-slug save still overwrites in place (post-PR-A, preserving `created_at`); the ┬¦2.6 "append guards accidental clobbers" idea is explicitly **deferred** to a separate decision and is *not* part of this change. `update_policy` is a *declaration* that drives the consolidation contract (see Changed) and future provenance work, not a new clobber-guard. New `tests/test_update_policy_axis.py` (real SQLite + tmp_path) covers round-trip + sticky carry-forward + param-overrides-sticky, the critical append-default-unchanged regression guard, CLI body parity, and `status` validation.

- **Telemetry recall-exclusion (ADR-015 ┬¦2.3, #398).** Memories carrying `metadata.kind: telemetry` in frontmatter (machine/monitor writes ŌĆö e.g. deterministic monitor probes) are now **hard-excluded** from default semantic recall (ADR-015 ┬¦6 Q3: hard-exclude, not down-rank) so monitoring churn no longer pollutes human recall. The filter lives in the shared store layer (`store.search` / `store.search_fts` / `store.list_recent`, inherited by `store.search_hybrid`) so every surface ŌĆö API, MCP, CLI, plugin ŌĆö inherits it (ADR-010 parity), the same pattern #371 used. Telemetry remains retrievable on demand via an explicit override: `include_telemetry: true` on the `/search` API, the `include_telemetry` MCP search arg, and `--include-telemetry` on the CLI. New `tests/test_telemetry_recall_exclusion.py` (real SQLite + tmp_path) covers default exclusion across vector/hybrid/FTS/recency, the explicit-override include path, a non-telemetry regression guard, and the end-to-end `/search` override.

- **Recall-write health check in `palinode doctor` (#371).** A new `fast`-tagged check, `recall_write_health`, reads `MAX(recall_count)` and the count of recalled rows directly from the `chunks` table. It reports **"recall metadata is being written"** when the loop is live, and **fails** ("loop appears severed") when chunks are indexed but `MAX(recall_count)=0` with zero rows ever recalled ŌĆö exactly the production state the 2026-05-29 audit surfaced. Passes cleanly on an empty DB (nothing to recall). Remediation points the operator at running a search and re-checking.

- **Source-quote integrity primitive (`palinode.core.quote_verify`, #459 / public #65 Q2).** The deterministic, offline core of the quote-drift check requested upstream: `normalize_quote` / `quote_hash` (smart-punctuation + whitespace folding for stable hashing), `verify_quote` (pure ŌĆö distinguishes an internally-tampered anchor from a drifted source), and `verify_memory_sources` (verifies a file's proposed `sources:` anchors, a clean no-op when none are present). Intentionally a plain integrity check ŌĆö no cryptographic signing or attestation ŌĆö so it has no dependency on the separate memory-attestation work. The frontmatter capture schema and the `doctor`/`lint` surface are deferred to #459 pending sign-off; this lands the verifier mechanism with full unit coverage (`tests/test_quote_verify.py`).

- **`update_policy` + `include_telemetry` as first-class plugin args (ADR-015 ┬¦5 parity, #480).** The OpenClaw plugin (`plugin/index.ts`) now exposes `update_policy` on `palinode_save` (enum `append`/`replace`, TypeBox `Union[Literal]`) and `include_telemetry` on `palinode_search` (boolean), forwarding both as top-level fields in the API request body. Also closes pre-existing parity gaps for `priority` (save) and `min_priority` (search). The parity-registry test (`npm test`) gains six new assertions: schema-declaration checks, enum shape checks, and fetch-intercept threading checks for both params.

- **`palinode init` SessionEnd floor hook gains three robustness gates (#378 floor/ceiling convergence).** The canonical session-end capture hook (`HOOK_SCRIPT` in `palinode/cli/init.py`, scaffolded by `palinode init`) folds in the gates the #471 prototype proved out, so the floor capture shipped to every user is no longer a thin best-effort POST: **(1) skip-if-`/wrap`-ran** ŌĆö if the transcript already holds a `palinode_session_end` tool call, the agent-authored "ceiling" capture is strictly richer than the deterministic floor, so the hook skips rather than writing a redundant duplicate (override with `PALINODE_HOOK_FORCE=1`); **(2) dry-run** ŌĆö `PALINODE_HOOK_DRYRUN=1` prints the payload it would POST and writes nothing, for verifying hook wiring without touching the API; **(3) fallback log** ŌĆö `curl -f` now fails on HTTP Ōēź400 too, and on any POST failure the payload is appended to `.claude/session-floor-fallback.jsonl` so a dropped capture is recoverable instead of lost. Always exits 0 (a floor-capture failure must never block session exit). New end-to-end gate tests in `tests/test_cli_init.py` run the rendered script under a stubbed `curl`/PATH. The local dev-only `session-end-floor.sh` hook stays as-is (it adds a transcript-derived summary + the `palinode session-end` CLI path) with a header documenting the convergence.

- **Logging convention doc (`docs/logging.md`, #337 Phase 2).** A short, prescriptive standard for log levels (pick by the reader's required action, not author sentiment ŌĆö a caught failure is never INFO; a silently-handled failure still gets one line), mandatory `key=value` structured fields per event class (model call / indexing failure / git op / config resolution), the inventory of failure modes that must surface to operators (defaults-loaded, degraded model call, non-zero subprocess, partial index write, skipped startup gate), and rate-limiting rules (startup-once not per-request; summaries over streams). Future logging fixes cite it; the scoped Phase-1 audit of the embed / save / git-persistence / config-discovery paths that grounds it lives internally (53 sites, dispositions, Phase-3 sequencing). Doc-only ŌĆö no behavior change.

- **Docs: Claude Code `auto`-permission-mode troubleshooting for `/wrap` / `/save` / `palinode_push` denials.** `docs/INSTALL-CLAUDE-CODE.md` gains a Troubleshooting entry explaining that in `auto` permission mode the remote-write MCP tools (`palinode_push`, `palinode_session_end`) can be denied by the permission classifier as un-allowlisted network egress ŌĆö even on a user-invoked `/wrap`. Documents the fix (add `mcp__palinode__palinode_push` / `_session_end` / `_save` / `_consolidate` to `permissions.allow`, global vs. project scope, `mcp__<server>__<tool>` naming) and the by-design caveat that an agent cannot add these allowlist rules to `settings.json` itself.

### Changed

- **README repositioned to lead with audit-grade / git-native provenance.** The headline moves from "memory substrate for AI agents" to "audit-grade memory for AI coding agents ŌĆö every fact you can `blame`, `diff`, `rollback`." Framing only; leads with provenance that already ships today (git lineage), no behavior change and no claim about not-yet-built capture (source-spans, extraction metadata).
- **Single memory-file mutation choke point (#564).** Every path that mutates a memory file now routes its write through one atomic primitive, `git_tools.write_memory_file(path, content)` (temp + fsync + rename, crash-safe), and its commit through `git_tools.commit_memory_file` / `commit_memory_files` (explicit-file staging). Migrated the formerly-scattered mechanisms onto them: the `/save` body write, the summary/description injectors, the consolidation executor's atomic write, ttl-archive, the consolidation runner's project/decision/status writes, the prompt-activate toggle, the openclaw/mem0/vault migration writers, the ingest pipeline, and the Obsidian wiki-footer sync. Pure substrate consolidation, no behaviour change ŌĆö concentrating writes and commits in one module gives the project one auditable path for memory mutations. The executor's `_atomic_write_text` is retained as a thin local alias delegating to the choke point.
- **Single op-parse/normalize module for consolidation (#555).** "Is this a well-formed op?" was answered defensively in four places, each re-deriving the `op`/`operation` and `reason`/`rationale` aliases and the `isinstance`/nested-list defense: `runner._consolidate_project` (extract + `json_repair` + filter), `executor.apply_operations` (`op.get("op","KEEP").upper()`), `runner._proposed_changes`, and `write_time._translate_ops`. These now share `palinode/consolidation/op_parse.py` ŌĆö `parse_operations(raw_text)` (a faithful extraction of the runner JSON-array parse + repair + filter, behaviour-preserving) plus canonical `op_kind(op)` / `op_reason(op)` accessors. A free win falls out: because `op_kind` coerces with `str()` before `.upper()`, a non-string `op` value no longer crashes on `.upper()` ŌĆö it's stringified, matches no op type, and is skipped, closing one of the #311 executor validation gaps (its previously-`xfail`'d test in `test_executor_ops_coverage` is now a real pass). New `tests/test_op_parse.py`. Pairs with #554: the proposer's output is now the parser's input.
- **Injectable proposeŌåÆapply seam for consolidation (#554).** Consolidation's nondeterministic half (the LLM that proposes ops) and deterministic half (`executor.apply_operations`) met inline, so the only way to test the runnerŌåÆexecutor path was to mock `_consolidate_project` wholesale ŌĆö skipping the real fact-extraction, prompt-building, JSON parse, executor application, and git commit. `_consolidate_project` / `_check_contradictions` now take an injectable `llm_fn` (`(system_prompt, user_prompt) -> (response_text, model_used)`, default = the live `_call_llm_with_fallback`), and `run_consolidation` / `run_nightly` thread it through. A fake adapter returning canned op-JSON drives the whole real pipeline deterministically. - **Session-end timeout raised to 90s on all three surfaces (#377).** The 30s CLI timeout and 10s hook curl timeout were too short for multi-decision payloads (BGE-M3 dedup embed + git commit on a remote private VPN host). All three surfaces now use a shared `SESSION_END_TIMEOUT_SECONDS = 90.0` constant from `palinode.core.defaults` (overridable via `PALINODE_HOOK_TIMEOUT` / `PALINODE_SESSION_END_TIMEOUT` env vars). CLI, MCP, and hook all import or reference the same source of truth; module-load assertion guards in `cli/_api.py` and `mcp.py` catch future drift at import time. Hook script `PALINODE_HOOK_TIMEOUT` default is 30s; `settings.json` runner timeout raised to 35s to stay ahead of curl. CLI now distinguishes `ReadTimeout` (slow API, recoverable) from `ConnectError` (API down) with separate error messages. Eight new regression tests in `tests/test_session_end_timeout.py`.
- **`/wrap` now pushes to remote before archiving (#353).** The `/wrap` slash command calls `palinode_push` (MCP tool) before `palinode_session_end`, so local commits are on the remote before the daily note is written and context is cleared. Skips silently when there are no unpushed commits or no remote is configured; surfaces a clear message on push failure so Paul can decide whether to proceed. `palinode init`'s `WRAP_COMMAND_BODY` is updated to match.
- **Embed path hardened against Ollama context-window mismatch (#335).** When Ollama returns a 200 OK with `{"error": "prompt is too long for max context"}` (its silent-truncation failure mode when `num_ctx` is too small), `_embed_local` now raises `EmbeddingContextError` ŌĆö a typed exception with `model`, `text_len`, and `ollama_message` attributes and a recovery hint pointing at the modelfile fix. Added a lazy preflight check (`check_model_context`) that queries `/api/show` on the first embed call and logs a WARNING if `num_ctx` is below 8192 (bge-m3's actual capability vs. Ollama's 4096 default ŌĆö the exact regression from 2026-05-04). Preflight runs once per process (thread-safe) and never blocks embed on failure.
- **`palinode_save` auto-description now has a hard timeout with graceful degrade (#336).** Previously the description-generation call to Ollama used a hardcoded 15 s timeout with no degrade path ŌĆö a cold-loading or overloaded Ollama would stall every `/save` for the full duration. Now: (1) `_generate_description` has a configurable timeout (`config.auto_summary.describe_timeout_seconds`, default 5 s, override via `PALINODE_DESCRIBE_TIMEOUT_SECONDS`) that returns a `_DESCRIPTION_DEFERRED` sentinel on `TimeoutException`; (2) the `/save` handler detects the sentinel, saves the file without a description, and includes `description_pending: true` in the response so callers know to expect a fill-in; (3) the watcher detects files with no `description` field after any file event and schedules a debounced retry via `/generate-summaries` (10 s window, accumulates rapid saves). The audit Q2 finding is also fixed: non-timeout description failures (e.g. `ConnectError`) were logged at `INFO`; they now log at `WARNING`.

### Added

- `tests/test_server_json_version_alignment.py` ŌĆö CI guard that fails if `server.json` top-level `version` or any `packages[*].version` drifts from `pyproject.toml[project.version]`. Silent drift was possible: `mcp-publisher publish` would ship the manifest with the wrong version pin with no runtime error to surface it. Four tests cover: top-level version match, per-package version match, JSON parseability, and TOML parseability ŌĆö fail loud on missing files or corrupt parse so no check silently passes (#313).
- **`palinode doctor` FTS5 sync check (#316).** New `fts5_sync` check detects row-count drift between the `chunks` source table and the `chunks_fts_docsize` FTS5 shadow table. A mid-write exception, schema migration, or crashed bulk-index run can leave FTS5 with fewer entries than `chunks`, causing BM25 keyword search to silently miss content. The check compares the two counts and fails with WARN + recovery hint ("run `palinode reindex`") when they diverge. Tagged `fast` ŌĆö pure SQLite read, no network I/O. ## [0.8.8] ŌĆö 2026-05-01

A pure-security release in response to the MCP Marketplace security audit.
Closes 27 of the 30 findings (5 critical, 8 high, 12 medium, 2 low). No
breaking changes; all hardening is additive. Local-first deployments keep
working without ceremony.

### Security

- **Optional bearer-token authentication on the API server.** Set
  `PALINODE_API_TOKEN` (or `PALINODE_API_TOKEN_FILE` for docker-secrets
  style deployments) to require `Authorization: Bearer <token>` on every
  request other than `/health` and `/health/watcher`. Off by default to
  preserve zero-friction local dev. The startup gate now refuses to launch
  when `PALINODE_API_BIND_INTENT=public` is set without a token, closing
  the public-bind hardening gap. Comparison uses `hmac.compare_digest` to
  remove the timing side-channel. The gate fires at module import so it
  triggers under `uvicorn palinode.api.server:app` (the canonical systemd
  ExecStart pattern), not just from `main()`.
- **Bearer-token authentication on the MCP HTTP transport (#289).** The
  MCP HTTP server (port 6341) now enforces the same `Authorization: Bearer
  <token>` requirement as the API server when `PALINODE_API_TOKEN` is set ŌĆö
  closing the gap where an authenticated API was reachable unauthenticated
  via the MCP HTTP surface. Token source is shared (`PALINODE_API_TOKEN` /
  `PALINODE_API_TOKEN_FILE`): one deployment, one secret. A separate
  `PALINODE_MCP_BIND_INTENT=public` gate (distinct from
  `PALINODE_API_BIND_INTENT`) refuses to start the MCP HTTP server when
  publicly bound without a token. The gate lives **inside `main_http()`**
  (not at module import) so importing `palinode.mcp` for the stdio
  transport ŌĆö which never calls `main_http()` ŌĆö does not trigger it.
  `/healthz` is exempt so uptime probes don't need the token. The shared
  auth primitives (`BearerAuthMiddleware`, `load_api_token`,
  `validate_auth_config`) are extracted into `palinode.core.auth` so
  neither server duplicates the security-sensitive comparison logic.
- **Dependency lower-bounds added** to exclude 15 known CVEs against
  `pyyaml`, `httpx`, `sqlite-vec`, `fastapi`, `uvicorn`, `pydantic`, and
  `mcp`. All currently-installed versions are already past the fix
  windows; this codifies that floor.
- **Path traversal hardening** in `_resolve_memory_path` migrated from
  `os.path.realpath`/`commonpath` to `pathlib.Path.resolve(strict=True)`
  + `Path.is_relative_to()`. Generic `"Invalid path"` returned to clients
  (no filesystem-info leak); original input logged at INFO for operators.
- **Streaming body-size enforcement.** New ASGI middleware counts bytes
  per chunk and rejects with HTTP 413 mid-stream. Closes the
  chunked-encoding bypass that the previous header-only check missed.
- **Wiki footer entity validation.** Slugs must match `^[A-Za-z0-9._-]+$`
  before being emitted inside `[[wikilinks]]`; malformed slugs are
  dropped with a warning rather than corrupting markdown.
- **LLM prompt content** in `_generate_description` and `_generate_summary`
  now wrapped in `<user_content>` delimited tags with neutralization of
  injected close tags. Best-effort defense against prompt-injection.
- **Subprocess argv-form audit** confirmed all `subprocess.run` calls in
  the codebase use list-of-args, never `shell=True`. AST-based test in
  `tests/test_subprocess_argv_form.py` fails CI on any future regression.
- **CORS wildcard rejection** at startup. `PALINODE_CORS_ORIGINS=*` now
  refuses to launch with a clear error; each origin must parse as a
  valid http/https URL with non-empty netloc.
- **Secret-redacting logging filter** prevents API keys (`sk-...`,
  `xoxb-...`, AWS keys) and basic-auth-in-URL credentials from leaking
  via `logger.exception()` tracebacks if a memory file contains them.
- **Rate limiter dict bounded.** `_rate_counters` now prunes expired
  entries on every check and caps at `PALINODE_RATE_LIMIT_MAX_KEYS`
  (default 10000) with oldest-window eviction. Prevents memory growth
  under varied-IP scans.
- **TOCTOU mitigation** in file reads. New `_open_memory_file_text()`
  helper uses `os.open(..., O_RDONLY | O_NOFOLLOW)` so symlink swaps
  raise `OSError` rather than silently following the link.
- **Narrowed broad `except Exception` blocks** at LLM, HTTP, and git
  call sites. Specific exception types (`httpx.HTTPError`, `OSError`,
  `json.JSONDecodeError`, etc.) make root-cause tracing easier and
  prevent operational errors from masking security signal.

### Fixed

  rationale rather than suppressed wholesale. `pip-audit` now runs (was
  blocked by the bandit failure in v0.8.5).

  `palinode_cluster_neighbors`, `palinode_topic_coverage`,

## [0.7.2] ŌĆö 2026-04-26

Bug-fix release with small UX additions. Brings the public repo up to date with two production-impacting fixes (MCP array coercion, SessionEnd hook reasoning) plus search filters and structured session metadata. v0.8.0 remains reserved for a later major feature release.

### Added

- **`/search` filters and recency-only mode (#141)** ŌĆö `types: [...]` and `since_days: N` are now honored. - **`palinode_save` `ps=true` shortcut (#136)** ŌĆö MCP parity with the CLI `--ps` flag. `palinode_save` accepts either an explicit `type` or `ps=true` (shorthand for ProjectSnapshot). Conflict between `ps=true` and a non-ProjectSnapshot `type` errors clearly rather than silently ignoring one.

### Fixed

- **MCP tolerates JSON-encoded array arguments (#147)** ŌĆö some MCP transports double-encode array args as JSON strings. Previous behavior rejected with Pydantic 422 ("expected array, received string") on `palinode_session_end` and `palinode_save`, and silently corrupted the `paths` filter on `palinode_diff` (`",".join(string)` was joining char-by-char). New `_coerce_str_array` helper decodes JSON-array strings while passing native lists through unchanged.
- **SessionEnd hook reason filter (#149)** ŌĆö the hook's README claimed a `clear` matcher that never existed in `settings.json`. Replaced with explicit script-side filtering via a new `PALINODE_HOOK_REASONS` env var. Default allowlist: `clear logout prompt_input_exit other`. Override to narrow (e.g. `"clear"`) or extend (`+resume`, `+bypass_permissions_disabled`).
- **Empty-transcript captures (#151)** ŌĆö SessionEnd hook was capturing 0-message sessions despite `PALINODE_HOOK_MIN_MESSAGES=3`. Fix: drop the `|| echo "0"` (grep always emits an integer regardless of exit), use `|| true` to absorb the non-zero, default to `0` if the pipeline produces nothing.
- **Test isolation** ŌĆö full-suite test runs no longer hit 5 failures in `test_write_time` due to scope-chain reload state pollution.

### Documentation

- **SSH keepalive options for remote-stdio MCP (`docs/MCP-SETUP.md`, `docs/INSTALL-CLAUDE-CODE.md`)** ŌĆö `ServerAliveInterval=30`, `ServerAliveCountMax=3`, and `TCPKeepAlive=yes` added to the example configs. ## [0.1.0] ŌĆö 2026-03-22

Initial public release. Minimum viable memory system.

### What worked at launch

- SQLite-vec vector store with BGE-M3 embeddings via Ollama
- File watcher daemon auto-indexing markdown on create/modify/delete
- FastAPI server with `/search`, `/save`, `/status`, `/reindex` endpoints
- Markdown parser with YAML frontmatter extraction and heading-level section chunking
- CLI with `search` and `stats` commands
- Plugin with core memory injection, topic-specific retrieval, and three tools (`palinode_search`, `palinode_save`, `palinode_status`)
- Session capture to daily notes on agent end
- Systemd user services for `palinode-api` (port 6340) and `palinode-watcher`

### Architecture decisions

| Decision | Choice | Rationale |
|---|---|---|
| Source of truth | Markdown files, git-versioned | Human-readable, survives everything |
| Vector store | SQLite-vec (embedded) | No server, matches file-based philosophy |
| Embeddings | BGE-M3 via Ollama | Local, private, strong on structured text |
| Prompts | Read from `specs/prompts/*.md` | Version-controlled, editable, diffable |

### Known limitations at 0.1.0

- No WAL mode on SQLite ŌĆö concurrent writes can lock
- Session capture wrote raw text to daily notes (no LLM extraction)
- No consolidation scheduler
- No entity linking
- No MCP server (plugin-only integration)

All of these are addressed in 0.5.0.
